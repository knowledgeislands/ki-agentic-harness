# ADR-KI-HARNESS-SKILLS-006: Six-cluster skill taxonomy and the implication graph

**Date:** 2026-07-09

## Context

The skill set grew to twenty without a stated organising model. The README described relationships in prose and ASCII art, and the only machine-readable substrate was `ki-repo`'s coverage array plus `ki-bootstrap`'s two-element baseline. Two things were missing: a named taxonomy that says what role each skill plays, and a single declared source for "linking skill X pulls in skill Y" that both the bootstrap chain and a user-facing dependency tree could derive from without drifting apart.

## Decision

The skills are organised into six clusters, and their relationships are declared in a machine-readable implication graph.

- **Kind axis (orthogonal to the clusters).** Every skill has a **kind** — **governance** or **process** — cutting across the six clusters. A **governance skill** holds a house standard and ships the universal modes (AUDIT/CONFORM/INIT/REFRESH + HELP, [ADR-KI-HARNESS-SKILLS-001](ADR-KI-HARNESS-SKILLS-001-canonical-modes.md)) and the four-file shape (`<domain>-standard.md`, `audit-rubric.md`, `sources.md`, a mechanical checker), which the `ki-skills` rubric enforces. A **process skill** drives an action or lifecycle rather than holding a standard: it is lightweight, may bundle a helper `scripts/` and a `references/` procedure (the "extra gubbins" a plain `.claude/commands/*.md` cannot carry), and is **exempt** from the governance four-file shape and universal modes — its mode count varies and it exposes HELP only optionally. Both kinds are dual-invocable (`/<name>` explicit and model-triggered by `description`); a process skill is a slash command with a script and references attached. The first process skills are **`ki-recap`** (a light one-mode skill — summarise / surface-outstanding / harvest-learnings over a live session, with an optional compress leg) and **`ki-plan`** (a richer lifecycle promoted from the former `/plan` command). `ki-plan` (singular verb — drive a plan) sits deliberately beside the governance skill `ki-plans` (plural — govern the class of plans): a governance/process pairing, not a collision. `ki-recap`'s compress leg writes a carry-forward digest only — true in-context window reduction is the native / PreCompact-hook path (`ki-tokenomics`), not something a skill can do. Kind is orthogonal to **install scope** (global vs project-local): both `ki-recap` and `ki-plan` are installed globally (`ki:skills:link:global`), for the same reason as the keystone `ki-bootstrap` — utility across any repo on the machine, not repo governance — and like `ki-bootstrap` they are therefore **never vendored or declared** in a repo's `.ki-config.toml` (no `[ki-plan]`/`[ki-recap]` table, ever); a governance skill's `[ki-<skill>]` table means "this repo is governed by this standard," which does not apply to a process skill with no standard to be governed by.
- **Clusters.** (1) **Keystone** — `ki-bootstrap`, `ki-repo`. (2) **Structure-independent governance** — `ki-authoring` (always linked, part of the `ki-repo` baseline) and `ki-engineering` (linked only where coverage-detected — see below). (3) **Repo-structure**, mutually exclusive, exactly one per repo — `ki-harness`, `ki-kb`, `ki-website`, `ki-mcp`, `ki-plugins`, `ki-tools`, `ki-homebrew-tap`. (4) **General governance** — `ki-agents`, `ki-skills`, `ki-decision-records`, `ki-feature-definitions`, `ki-handoffs`, `ki-plans`. (5) **Implied families** — the KB family under `ki-kb`, the website family under `ki-website`. (6) **Environment**, governing the user's machine rather than a repo — `ki-binding`, `ki-housekeeping`, `ki-tokenomics`.
- **Implication graph.** Each SKILL.md declares an `implies:` frontmatter list — the single source from which `skills/ki-bootstrap/scripts/skill-graph.ts` generates both the bootstrap chain and the dependency tree (`ki:skills:graph`), validated in `ki:verify`.
- **Implied vs coverage-detected.** Not every cluster-2 skill is implied by the keystone. `ki-authoring` is universal — governing Markdown/TOML style applies to any repo, so `ki-repo` implies it and it is part of the baseline vendored/linked everywhere. `ki-engineering` governs the TS/Bun toolchain, which only exists where a `package.json` does — so it is **coverage-detected**, not implied: `ki-repo`'s coverage cascade expects `[ki-engineering]` to be declared by any repo with a `package.json`, and only then does it get vendored/linked. A non-code repo (dotfiles, a KB, the homebrew tap) never carries the edge at all, rather than carrying it and having the checker NA out.
- **Mutual exclusion.** A repo carries exactly one repo-structure skill; `ki-repo`'s coverage cascade enforces it mechanically.
- **Canonical names.** Each skill's directory name is its canonical name. `ki-housekeeping` broadens the former memory skill into machine-state hygiene generally, and `ki-feature-definitions` is the general-governance skill for feature specs.

## Consequences

- A new skill is placed by choosing its cluster; its edges are declared in one greppable place.
- The bootstrap chain and the user-guide tree cannot drift — both derive from `implies:`, and a broken edge fails CI.
- Cluster 6 gives the machine/environment skills a home the earlier repo-only taxonomy lacked.
- The generated plugin-marketplace repo (`knowledgeislands/ki-plugins`, `ADR-KI-HARNESS-002`) is a further repo shape; `ki-plugins` governs its on-disk projection so it is no longer a repo carrying zero repo-structure skills. Its **generation and cross-surface enablement** stay with the environment skill `ki-binding` (they govern the machine, not the repo) — the split keeps each cluster's defining property intact.

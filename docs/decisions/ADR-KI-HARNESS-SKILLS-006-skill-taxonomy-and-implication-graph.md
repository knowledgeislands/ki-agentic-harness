# ADR-KI-HARNESS-SKILLS-006: Six-cluster skill taxonomy and the implication graph

**Date:** 2026-07-09

## Context

The skill set grew to twenty without a stated organising model. The README described relationships in prose and ASCII art, and the only machine-readable substrate was `ki-repo`'s coverage array plus `ki-bootstrap`'s two-element baseline. Two things were missing: a named taxonomy that says what role each skill plays, and a single declared source for "linking skill X pulls in skill Y" that both the bootstrap chain and a user-facing dependency tree could derive from without drifting apart.

## Decision

The skills are organised into six clusters, and their relationships are declared in a machine-readable implication graph.

- **Kind axis (orthogonal to the clusters).** Every skill has a **kind** — **governance** or **process** — cutting across the six clusters. A **governance skill** holds a house standard and ships the universal modes (AUDIT/CONFORM/EDUCATE/REFRESH + HELP, [ADR-KI-HARNESS-SKILLS-001](ADR-KI-HARNESS-SKILLS-001-canonical-modes.md)) and the four-file shape (`<domain>standards.md`, `rubric.md`, `sources.md`, a mechanical checker), which the `ki-skills` rubric enforces. A **process skill** drives an action or lifecycle rather than holding a standard: it is lightweight, may bundle a helper `scripts/` and a `references/` procedure (the "extra gubbins" a plain `.claude/commands/*.md` cannot carry), and is **exempt** from the governance four-file shape and universal modes — its mode count varies and it exposes HELP only optionally. Both kinds are dual-invocable (`/<name>` explicit and model-triggered by `description`); a process skill is a slash command with a script and references attached. The process skills are **`ki-recap`** (a light one-mode skill — summarise / surface-outstanding / harvest-learnings over a live session, with an optional compress leg and an optional current-session handoff to `ki-next`), **`ki-next`** (the confirmation-gated non-KB next-work selector, which may invoke `ki-plan` after re-grounding the roadmap), **`ki-plan`** (the non-KB thematic-plan lifecycle paired with `ki-repo-roadmap`), and **`ki-delegate`** (decomposes and fans a multi-task effort out across agent/model tiers). `ki-recap`'s compress leg writes a carry-forward digest only — true in-context window reduction is the native / PreCompact-hook path (`ki-tokenomics`), not something a skill can do. Kind is orthogonal to **install scope** (global vs project-local): `/harness/install` installs the process skills globally, for the same reason as the keystone `ki-bootstrap` — utility across any repo on the machine, not repo governance — and like `ki-bootstrap` they are therefore **never vendored or declared** in a repo's `.ki-config.toml` (no `[ki-plan]`/`[ki-recap]`/`[ki-next]`/`[ki-delegate]` table, ever); a governance skill's `[ki-<skill>]` table means "this repo is governed by this standard," which does not apply to a process skill with no standard to be governed by.
- **Clusters.** (1) **Keystone** — `ki-bootstrap`, `ki-repo`. (2) **Foundations** (structure-independent governance) — `ki-authoring` and `ki-engineering`. (3) **Repo-structure**, mutually exclusive, exactly one per repo — `ki-harness`, `ki-kb`, `ki-website`, `ki-mcp`, `ki-plugins`, `ki-specifications`, `ki-tools`, `ki-homebrew-tap`, `ki-dotfiles-chezmoi`. `ki-specifications` begins as a deliberately minimal structural floor: a keyless marker and the established top-level authority/supporting areas, with deeper rules factored in only after they stabilise. (4) **General governance** — `ki-agents`, `ki-skills`, `ki-decision-records`, `ki-feature-definitions`, `ki-handoffs`, `ki-repo-roadmap` (non-KB repositories only). (5) **Dependent families** — `ki-kb-streams`, `ki-kb-activities`, `ki-kb-live-artifacts` (the KB family required by `ki-kb`), `ki-website-cloudflare` (the website family required by `ki-website`), `ki-binding-chezmoi` (the binding family required by `ki-binding`). (6) **Environment**, governing the user's machine rather than a repo — `ki-binding`, `ki-housekeeping`, `ki-tokenomics`.
- **Physical layout.** The six clusters are also physically encoded: every skill lives at `skills/<cluster>/<name>` (cluster names kebab-case per the list above — `keystone`, `foundations`, `repo-structure`, `general-governance`, `implied-families`, `environment`), not the earlier flat `skills/<name>`. The four process-kind skills live under a seventh, **physical-only** `skills/process/` directory — this is not a seventh conceptual cluster; kind remains the orthogonal axis, and `process/` simply gives it a physical home the way the six clusters give the cluster axis one. The cluster (and `process/`) subfolder is a filesystem grouping only: the canonical skill **name** referenced everywhere else (`.ki-config.toml`, `depends-on:` frontmatter, package.json script bodies) stays the bare `ki-<name>`, never the path — `skills/keystone/ki-bootstrap/scripts/resolve.ts` resolves a bare name to its on-disk cluster path via a memoized index, so no consumer outside that resolver needs to know which cluster it lives in.
- **Dependency graph.** Each SKILL.md declares a `depends-on:` frontmatter list — the single source from which `skills/keystone/ki-bootstrap/scripts/skill-graph.ts` generates the dependency tree (`ki:skills:graph`) and validates graph integrity. A dependency identifies a necessary governance capability only: it neither adds hidden coverage nor establishes mode-execution order. A repo declaring a skill MUST explicitly declare each dependency in `.ki-config.toml`; bootstrap and project publication reject a missing declaration before mutation.
- **Universal vs coverage-detected.** `ki-authoring` is a universal requirement of `ki-repo`, declared by its `depends-on:` edge and explicitly present in each repo's configuration. `ki-engineering` governs the TS/Bun toolchain, which only exists where a `package.json` does — it remains **coverage-detected**: `ki-repo`'s coverage cascade expects `[ki-engineering]` to be declared by any repo with a `package.json`, and only then does it get vendored/linked. A non-code repo (dotfiles, a KB, the homebrew tap) carries neither that table nor an inactive edge.
- **Mutual exclusion.** A repo carries exactly one repo-structure skill; `ki-repo`'s coverage cascade enforces it mechanically.
- **Canonical names.** Each skill's directory name is its canonical name. `ki-housekeeping` broadens the former memory skill into machine-state hygiene generally, and `ki-feature-definitions` is the general-governance skill for feature specs.

## Consequences

- A new skill is placed by choosing its cluster; its edges are declared in one greppable place.
- The user-guide tree cannot drift from the dependency graph; a broken edge fails CI. Bootstrap validates dependencies against explicit repository declarations rather than treating the graph as a coverage-expansion mechanism.
- Cluster 6 gives the machine/environment skills a home the earlier repo-only taxonomy lacked.
- The generated plugin-marketplace repo (`knowledgeislands/ki-plugins`, `ADR-KI-HARNESS-002`) is a further repo shape; `ki-plugins` governs its on-disk projection so it is no longer a repo carrying zero repo-structure skills. Its **generation and cross-surface enablement** stay with the environment skill `ki-binding` (they govern the machine, not the repo) — the split keeps each cluster's defining property intact.
- Runtime-adapter concerns (multi-runtime support) are not a new taxonomy cluster or axis — they live inside individual skills' standards and each repo's `.ki-config.toml`, not a seventh cluster.

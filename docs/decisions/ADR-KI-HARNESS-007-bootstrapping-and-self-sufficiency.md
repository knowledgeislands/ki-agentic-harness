# ADR-KI-HARNESS-007: The bootstrapping chain and the self-sufficiency contract

**Date:** 2026-07-09

## Context

A repo is brought under Knowledge Islands governance in two situations the earlier "install the skills, then run them" model served poorly: a greenfield repo with nothing installed, and a legacy Knowledge Islands repo carrying old formats that must be migrated. Requiring the full skill set to be installed first makes onboarding heavy and couples every repo to a global install. The mechanical-first stance ([ADR-KI-HARNESS-004](ADR-KI-HARNESS-004-mechanical-first-progressive-enhancement.md)) already lets the scripts run without an LLM; the remaining step is to let them run without the skill installed at all.

## Decision

Governance bootstraps through a single chain, runnable from the remote source with no skill installed locally.

- The bootstrap is the minimal chain-starter — kept globally installed only to keep the global surface tiny. It always pulls in the repo standard; from there each skill's INIT (its `scripts/init.ts` — the mode's mechanical half carries the mode's verb, like `audit-*.ts` and `conform-*.ts`) declares and triggers the skills it `implies:`. The name "bootstrap" belongs to exactly one thing: the chain engine inside `ki-bootstrap`.
- INIT satisfies the **self-sufficiency contract**, and does so **without requiring the target to have a `package.json`** — a non-code repo (dotfiles, KB, tap) is bootstrapped the same way. Into the target repo it (1) **vendors copies** of each skill's mechanical scripts (copies, not symlinks, so they run standalone) under a dot-prefixed, generated-not-authored `.ki-meta/skills/<skill>/` — kept off the repo's own `scripts/` — (2) writes a `.ki-meta/bin/aggregate.ts` runner that discovers those copies on the filesystem (it does **not** read `package.json`) and fans out across them for a given verb, and (3) writes an executable `.ki-meta/bin/ki-audit` wrapper — the `package.json`-free entry point. Where the target **does** have a `package.json`, INIT additionally installs that skill's `ki:<suffix>:{init,audit,conform}` keys and the repo-wide `ki:audit` / `ki:conform` / `ki:init` aggregates as convenience aliases over the same runner. The npm-script wiring is thus an _additive_ code-repo convenience, not the contract. Stated as a principle: **the vendored `.ki-meta/skills/<skill>/<verb>.ts` script is the canonical mechanical unit, and every `ki:*` `package.json` key is syntactic sugar over the identical invocation of it** — the two never diverge, and a repo with no manifest loses nothing.
- The new / legacy / tracking modes are aggressiveness flags over this one chain, not a separate orchestrator: new = INIT; legacy = INIT + full CONFORM (the migration path); tracking = AUDIT + mechanical CONFORM.
- A confirmed Knowledge Islands repo that carries no self-check runner (`.ki-meta/bin/ki-audit` or `.ki-meta/bin/aggregate.ts`) is a **FAIL** in the repo standard — a `.ki-config.toml` marker without a way to run the checks is incomplete. There is no WARN ramp: the fleet re-bootstraps to the new form.

## Consequences

- After bootstrap a repo governs itself with `./.ki-meta/bin/ki-audit` (or `bun run ki:audit` where a `package.json` exists) and zero skills installed.
- **Which modes vendor, and why they differ.** The aggregate runner accepts `audit | conform | init`, but the modes are not symmetric in what they can vendor, and a governed repo will typically show `audit` alone. This is a property of each mode's mechanical half, not a limitation of the mechanism:
  - **AUDIT** vendors for every skill — each governance skill ships exactly one standalone `audit-*.ts` / `lint-*.ts` checker, the universal deterministic unit, re-runnable against the target.
  - **CONFORM** vendors only where the skill has authored a `conform-*.ts`. The mechanism carries it identically to audit; most skills have simply not written one yet, so conform is present for a minority (e.g. `ki-repo`, `ki-binding`) and absent elsewhere.
  - **INIT** vendors nothing per skill, because there is no per-skill re-runnable init check to fan out over — INIT's mechanical half delegates to the one-shot bootstrap engine that _produces_ `.ki-meta`. The aggregate `init` verb is instead the **local re-sync prompt**: it invokes the vendored `.ki-meta/bin/ki-init` wrapper, which re-runs the remote chain at the ref recorded in the vendoring manifest (with a flag to move to a newer ref). Re-running init is re-bootstrapping — idempotent, network-requiring, and driven by the staleness signal the audit raises.
  - **HELP** does not vendor: its mechanical half is the shared `skill-help.ts` renderer, which reads a `SKILL.md`'s declared headings — and SKILL.md files are not present in a bootstrapped target, so it cannot run standalone there the way a checker can. Vendoring HELP into a target (a metadata manifest, or a rendered snapshot) is an open design question, tracked on the ROADMAP.
- A greenfield repo and a legacy Knowledge Islands repo migrate by the same remote-run flow; the only difference is the aggressiveness flag.
- **Drift is detected mechanically, advised by CONFORM, and repaired by re-running INIT.** At vendor time the engine stamps a manifest (`.ki-meta/manifest.json`: the harness ref plus a hash per vendored file). The repo standard's checker verifies the vendored copies against that manifest **offline** (integrity — a tampered or partial vendor FAILs) and, when the source is reachable, reports staleness against the ref. CONFORM never re-syncs the copies itself — in a bootstrapped-only repo the local copies are the only source present, so re-syncing from them would be circular; it prints the advisory instead ("vendored scripts stale — re-run INIT"). The repair is an idempotent re-run of the chain (via `ki-init` or the remote one-liner), so a REFRESH that changes a checker propagates on the next re-bootstrap.
- The remote-run transport is a documented `bun run <raw-github-url>` one-liner against the harness on GitHub — no publish step, pinned to a ref (branch or tag). A skill's INIT is reached as `bun run https://raw.githubusercontent.com/knowledgeislands/ki-agentic-harness/<ref>/skills/<skill>/scripts/init.ts <target>`; the bootstrap starts the chain the same way.
- The self-sufficiency mechanics live **once**, in the bootstrap's chain engine (`scripts/bootstrap.ts`): it discovers each skill's checker/conform scripts by naming convention, resolves the `implies:` closure, vendors the copies into `.ki-meta/`, writes the `.ki-meta/bin/aggregate.ts` runner and `.ki-meta/bin/ki-audit` wrapper, and (where a `package.json` exists) wires the convenience keys. Each skill's `scripts/init.ts` is a **thin delegator** that execs that engine with itself as an explicit `--seed`. Delegating by subprocess is composition — running a sibling in sequence — not a cross-skill import, so every skill keeps its INIT mechanical half without duplicating the engine and stays valid standalone (the standalone-validity requirement of [ADR-KI-HARNESS-SKILLS-004](ADR-KI-HARNESS-SKILLS-004-skills-valid-standalone.md), earlier in the reading order).

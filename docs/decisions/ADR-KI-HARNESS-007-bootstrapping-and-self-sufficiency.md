# ADR-KI-HARNESS-007: The bootstrapping chain and the self-sufficiency contract

**Date:** 2026-07-09

## Context

A repo is brought under Knowledge Islands governance in two situations the earlier "install the skills, then run them" model served poorly: a greenfield repo with nothing installed, and a legacy Knowledge Islands repo carrying old formats that must be migrated. Requiring the full skill set to be installed first makes onboarding heavy and couples every repo to a global install. The mechanical-first stance ([ADR-KI-HARNESS-004](ADR-KI-HARNESS-004-mechanical-first-progressive-enhancement.md)) already lets the scripts run without an LLM; the remaining step is to let them run without the skill installed at all.

## Decision

Governance bootstraps through a single chain, runnable from the remote source with no skill installed locally.

- The bootstrap is the minimal chain-starter — kept globally installed only to keep the global surface tiny. It always pulls in the repo standard; from there each skill's INIT (its `scripts/init.ts` — the mode's mechanical half carries the mode's verb, like `audit-*.ts` and `conform-*.ts`) declares and triggers the skills it `implies:`. The name "bootstrap" belongs to exactly one thing: the chain engine inside `ki-bootstrap`.
- INIT satisfies the **self-sufficiency contract**, and does so **without requiring — or ever touching — the target's `package.json`**; a non-code repo (dotfiles, KB, tap) is bootstrapped exactly like a code repo. Bootstrap's one job is to build `.ki-meta/`: for every skill in the resolved set it (1) **vendors copies** of the skill's declared mechanical unit (copies, not symlinks, so they run standalone) plus a rendered **HELP snapshot** under a dot-prefixed, generated-not-authored `.ki-meta/skills/<skill>/` — kept off the repo's own `scripts/` — and (2) writes the four `package.json`-free entry points `.ki-meta/bin/{ki-audit, ki-conform, ki-init, ki-help}` over a `.ki-meta/bin/aggregate.ts` runner that discovers those copies on the filesystem (it does **not** read `package.json`) and fans out across them for a given verb. Stated as a principle: **the vendored `.ki-meta/skills/<skill>/<verb>.ts` script is the canonical mechanical unit, and every `ki:*` `package.json` key is syntactic sugar over the identical invocation of it** — the two never diverge, and a repo with no manifest loses nothing. The `ki:*` convenience keys a code repo may carry are **not bootstrap's to write** — `ki-engineering` wires them, as sugar over these same bins, when it comes online for that repo (see the Consequences).
- **Re-running INIT is the single, idempotent way to bring a target up to date** — there are no `--new` / `--legacy` / `--tracking` aggressiveness modes. Re-running the chain at a ref re-vendors and re-stamps the manifest, reproducing byte-identical output; drift is detected by the audit and repaired by exactly this re-run (below).
- A confirmed Knowledge Islands repo that carries no self-check runner (`.ki-meta/bin/ki-audit` or `.ki-meta/bin/aggregate.ts`) is a **FAIL** in the repo standard — a `.ki-config.toml` marker without a way to run the checks is incomplete. There is no WARN ramp: the fleet re-bootstraps to the new form.

## Consequences

- After bootstrap a repo governs itself with the four `./.ki-meta/bin/` wrappers and zero skills installed. Where `ki-engineering` has come online for a code repo, `bun run ki:audit` and friends alias them — but that aliasing is engineering's, not bootstrap's.
- **What each aggregate verb vendors, and why they differ.** The runner accepts `audit | conform | init | help`; the verbs are not symmetric in what they can vendor, though every governed repo now carries all four bins. This is a property of each verb's mechanical half:
  - **AUDIT** vendors for every skill — each governance skill ships exactly one standalone `audit-*.ts` / `lint-*.ts` checker, the universal deterministic unit, re-runnable against the target.
  - **CONFORM** vendors only where the skill has authored a `conform-*.ts`. The mechanism carries it identically to audit; most skills have simply not written one yet, so conform is present for a minority (e.g. `ki-repo`, `ki-binding`) and absent elsewhere. The `ki-conform` bin runs whatever conform checkers were vendored.
  - **INIT** vendors nothing per skill, because there is no per-skill re-runnable init check to fan out over — INIT's mechanical half delegates to the one-shot bootstrap engine that _produces_ `.ki-meta`. The aggregate `init` verb is instead the **local re-sync prompt**: it invokes the vendored `.ki-meta/bin/ki-init` wrapper, which re-runs the chain at the ref recorded in the vendoring manifest (with a flag to move to a newer ref). Re-running init is re-bootstrapping — idempotent, network-requiring, and driven by the staleness signal the audit raises.
  - **HELP** vendors a **rendered snapshot** per skill. At vendor time the engine has every `SKILL.md`, so it runs the shared `skill-help.ts` renderer once per skill and writes the result to `.ki-meta/skills/<skill>/help.md`; the `ki-help` bin is **pure bash** over those snapshots, readable even where bun is absent. This resolves the former open question (a target has no `SKILL.md`, so the renderer cannot run there live) by rendering at the one moment the sources are present; the snapshot is manifest-hashed like every other vendored file, so drift is caught the same way.
- A greenfield repo and an existing Knowledge Islands repo onboard by the same remote-run flow, and re-running it is the update path — there is no separate migration mode.
- **Drift is detected mechanically, advised by CONFORM, and repaired by re-running INIT.** At vendor time the engine stamps a manifest (`.ki-meta/manifest.json`: the harness ref plus a hash per vendored file). The repo standard's checker verifies the vendored copies against that manifest **offline** (integrity — a tampered or partial vendor FAILs) and, when the source is reachable, reports staleness against the ref. CONFORM never re-syncs the copies itself — in a bootstrapped-only repo the local copies are the only source present, so re-syncing from them would be circular; it prints the advisory instead ("vendored scripts stale — re-run INIT"). The repair is an idempotent re-run of the chain (via `ki-init` or the remote one-liner), so a REFRESH that changes a checker propagates on the next re-bootstrap.
- The remote-run transport is a documented zero-install one-liner against the harness on GitHub — no publish step, pinned to a ref (branch or tag), in the `curl | bash` installer idiom: `curl -fsSL https://raw.githubusercontent.com/knowledgeislands/ki-agentic-harness/<ref>/skills/ki-bootstrap/scripts/bootstrap.sh | bash -s -- <target> --ref <ref>`. The entry point assumes only bash/curl/tar: `bootstrap.sh` fetches the **source tarball** at the ref, extracts it to a temp dir, and runs the chain engine from that tree (Bun cannot execute a module over HTTP, and bun is the mechanical layer's _runtime_, not the entry point's — a missing bun fails fast with the install instruction rather than being installed silently). A skill's INIT is reached through `skills/<skill>/scripts/init.ts` in the extracted tree. The vendored `ki-init` wrapper pipes the same script at the manifest's recorded ref, so the transport is implemented exactly once; `--ref` is how a tarball extract (which has no `.git`) still stamps the manifest with the ref it ran at. Where bun is already installed, the harness `package.json` also declares the engine as its `bin`, so the `bunx` idiom works with no shell script in between: `bunx github:knowledgeislands/ki-agentic-harness#<ref> <target> --ref <ref>` — pinned to a sha, because bunx caches floating git refs.
- The self-sufficiency mechanics live **once**, in the bootstrap's chain engine (`scripts/bootstrap.ts` in `ki-bootstrap`): it reads each skill's **declared vendorable unit(s)**, resolves the `implies:` closure, vendors into `.ki-meta/` (checker/command units plus the per-skill HELP snapshot), stamps the manifest, and writes the `.ki-meta/bin/aggregate.ts` runner and the four `.ki-meta/bin/{ki-audit, ki-conform, ki-init, ki-help}` wrappers. It writes **no** `package.json` — that is `ki-engineering`'s to wire as sugar over these bins.
- **Each skill declares its vendorable unit(s) in its `SKILL.md` frontmatter, beside `implies:`** (the `vendors:` field); the central engine reads the declarations and vendors — per-skill declaration, central execution. A declaration is either a **checker file** (vendored as a copy) or a **command** (vendored as a generated thin wrapper script), so even a skill whose mechanical gate is a shared command — `ki-authoring` delegating to the Markdown toolchain, say — yields a unit runnable in a non-engineering, no-`package.json` repo via `./.ki-meta/bin/ki-audit`. The two per-skill facts the resolver needs — dependency closure and mechanical unit — thus live together in the skill's own file, keeping each skill self-describing and valid standalone; a target's `.ki-config.toml` continues to declare only _which_ skills apply, never what they contribute. During migration, a skill without a declaration falls back to the old filename-convention discovery with a **WARN** naming the missing declaration — never a hard fail mid-fleet. Each skill's `scripts/init.ts` is a **thin delegator** that execs that engine with itself as an explicit `--seed`. Delegating by subprocess is composition — running a sibling in sequence — not a cross-skill import, so every skill keeps its INIT mechanical half without duplicating the engine and stays valid standalone (the standalone-validity requirement of [ADR-KI-HARNESS-SKILLS-004](ADR-KI-HARNESS-SKILLS-004-skills-valid-standalone.md), earlier in the reading order).

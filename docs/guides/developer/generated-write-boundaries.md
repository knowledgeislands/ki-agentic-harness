# Generated write boundaries

Knowledge Islands keeps authored source, durable generated governance, runtime deployment payloads, and local-development links separate.

The boundary matters because each surface has a different owner and recovery path.

Do not repair one surface by copying or deleting files from another.

## The surfaces

- **Harness source** — `skills/`, `agents/`, guides, and tests are authored and committed in `ki-agentic-harness`.
- **Vendored governance** — `.ki-meta/` is generated by `ki-bootstrap` EDUCATE from a repository's declared coverage. It contains the durable mechanical checker copies, rendered HELP, manifest, and `ki-{audit,conform,educate,help}` bins a repository needs to govern itself without installed skills. It is a committed generated artifact: change its source, re-vendor it, then commit both together.
- **Runtime skill payloads** — `.claude/skills/` and `.agents/skills/` are generated regular-file copies for ordinary repositories and source links for a harness. They let an interactive session discover the repository's declared skills, but are gitignored and never committed.
- **Runtime agent payloads** — `.claude/agents/` is generated, gitignored state. Bootstrap leaves it untouched. Codex agent publication is not yet wired.
- **Package command convenience** — `ki-engineering` owns any `package.json` `ki:*` entries. They are optional aliases over `.ki-meta/bin/`; `ki-bootstrap` never writes them.

## Normal operation

Bootstrap is the normal writer.

It resolves `.ki-config.toml`, vendors `.ki-meta/`, and publishes runtime skill payloads for the selected runtimes.

Run it again to move a repository to a newer harness revision or restore a stale vendored surface:

```bash
./.ki-meta/bin/ki-educate
```

For the remote and fleet forms, see [Bootstrap a repository](../user-guide/onboarding.md).

Runtime payloads are reproduced from declarations, not committed files.

The publisher writes the matching `.gitignore` entries, so a fresh clone recreates the payloads by re-running bootstrap.

## Harness-local links

Bootstrap maintains this checkout's generated runtime skill links to its canonical sources:

```bash
bun run ki:educate
```

This applies only to a harness target, links only known runtime skill payloads beneath the supported runtime roots, and never changes `.ki-meta/`.

It lets local skill edits take effect immediately; consumer bootstrap remains copy-only.

See [Local skill linking](linking-skills.md) for runtime-specific paths, previews, and removal.

## Preservation and recovery

The publishers refuse unsafe parents and preserve unfamiliar, altered, or invalidly marked runtime payloads instead of overwriting them.

Treat that result as a migration diagnostic, not an invitation to force-delete files:

1. Inspect the local material and decide whether it is an intentional development link or user-owned content.
2. Preserve or move user-owned material deliberately.
3. Re-run the appropriate normal publisher only after the destination is known to be generated state.

Use the owner rather than guessing:

- `.ki-meta/` drift or an out-of-date harness revision → re-run **bootstrap EDUCATE**.
- Missing or ordinary runtime skill copies → run normal **repository bootstrap**.
- Harness project skill links → normal EDUCATE maintains them; ordinary EDUCATE restores copies.
- `package.json` command drift → run **ki-engineering** CONFORM.
- A user-global hook payload or runtime setting → use the applicable **user-environment manager**; repository bootstrap must not write it.

The normative rules and checker criteria remain with the `ki-bootstrap` standard.

This guide is the operational map for choosing the right writer and avoiding destructive cross-surface repairs.

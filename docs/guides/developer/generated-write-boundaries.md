# Generated write boundaries

Knowledge Islands keeps authored source, durable generated governance, runtime deployment payloads, and local-development links separate.

The boundary matters because each surface has a different owner and recovery path.

Do not repair one surface by copying or deleting files from another.

## The surfaces

- **Harness source** — `skills/`, `agents/`, guides, and tests are authored and committed in `ki-agentic-harness`.
- **Vendored governance** — `.ki/` is generated by `ki-bootstrap` EDUCATE from a repository's declared coverage. An ordinary repository receives durable mechanical checker copies, rendered HELP, a manifest, and `ki-{audit,conform,educate,help}` bins. A physical source harness instead gets manifest-proven relative links for canonical `skills/`, `agents/`, educator, support-module, and direct-checker material; its HELP, launchers, bins, and manifest stay regular generated files. For live source links, verify with `bun run ki:bootstrap:audit`; re-run bootstrap only when generated material changes. It remains a committed generated artifact.
- **Runtime skill payloads** — `.claude/skills/` and `.agents/skills/` are generated regular-file copies for ordinary repositories and source links for a harness. A repo-local `.ki/self/skill/` is the exception: it is committed source, projected into each declared runtime by a relative directory link. The runtime roots remain gitignored and never carry a committed `ki-self` copy.
- **Source support payloads** — a harness's frontmatter-declared `skills/*/scripts/vendored/` payloads are links to their canonical provider modules. Bootstrap dereferences those links for ordinary consumers; a source harness retains canonical links within its own `.ki/bootstrap/` surface.
- **Runtime agent payloads** — `.claude/agents/` is generated, gitignored state. Bootstrap leaves it untouched. Codex agent publication is not yet wired.
- **Package command convenience** — `ki-engineering` owns any `package.json` `ki:*` entries. They are optional aliases over `.ki/bin/`; `ki-bootstrap` never writes them.

## Normal operation

Bootstrap is the normal writer.

It resolves `.ki-config.toml`, vendors `.ki/`, and publishes runtime skill payloads for the selected runtimes.

Run it again to move a repository to a newer harness revision or restore a stale vendored surface:

```bash
./.ki/bin/ki-educate
```

For the remote and fleet forms, see [Bootstrap a repository](../user-guide/onboarding.md).

Runtime payloads are reproduced from declarations, not committed files. A local `ki-self` is authored once at `.ki/self/skill/SKILL.md`; bootstrap projects it into each declared runtime.

The publisher writes the matching `.gitignore` entries, so a fresh clone recreates the payloads by re-running bootstrap.

## Harness-local links

Bootstrap maintains this checkout's generated runtime skill and `.ki/bootstrap/` source links to its canonical sources:

```bash
bun run ki:educate
```

This applies only to a physical same-root harness target, links only manifest-declared canonical material beneath its own `skills/` and `agents/` trees, and never links an ordinary consumer, archive, or release payload.

It lets local skill edits take effect immediately; consumer bootstrap remains copy-only.

See [Local skill linking](linking-skills.md) for runtime-specific paths, previews, and removal.

## Preservation and recovery

The publishers refuse unsafe parents and preserve unfamiliar, altered, or invalidly marked runtime payloads instead of overwriting them. They migrate legacy `ki-self` copies only when they are valid and identical; divergent copies remain untouched for deliberate reconciliation.

Treat that result as a migration diagnostic, not an invitation to force-delete files:

For a source-owned reset of only proven generated state, run `bun skills/keystone/ki-bootstrap/scripts/clean.ts <target> --dry-run` first. CLEAN removes an exact manifest-owned `.ki/` tree and unchanged marker-owned regular runtime copies; it preserves every unproven path and refuses unsafe metadata.

1. Inspect the local material and decide whether it is an intentional development link or user-owned content.
2. Preserve or move user-owned material deliberately.
3. Re-run the appropriate normal publisher only after the destination is known to be generated state.

Use the owner rather than guessing:

- `.ki/` drift or an out-of-date harness revision → re-run **bootstrap EDUCATE**.
- Missing or ordinary runtime skill copies → run normal **repository bootstrap**.
- Harness project skill links → normal EDUCATE maintains them; ordinary EDUCATE restores copies.
- `package.json` command drift → run **ki-engineering** CONFORM.
- A user-global hook payload or runtime setting → use the applicable **user-environment manager**; repository bootstrap must not write it.

The normative rules and checker criteria remain with the `ki-bootstrap` standard.

This guide is the operational map for choosing the right writer and avoiding destructive cross-surface repairs.

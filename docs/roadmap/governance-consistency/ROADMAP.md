### Normalise mode vocabulary and mode-heading structure across the skills

**Sequence.** The convention-placement model + knowledge-promotion loop is the meta-item — it names the layers the concrete items here are instances of (`CLAUDE.md` → skill, the `docs/guides/agents/` staging area, the git-write-safety skill extraction), so let those inform it, but it must **not** front-run the multi-machine state-migration item (Soon) or the KI-vs-portable split (Future), which own the moves it only maps. The mode-vocabulary/heading normalisation is independent.

Building the HELP mode ([ADR-KI-HARNESS-SKILLS-001](../../../docs/decisions/ADR-KI-HARNESS-SKILLS-001-canonical-modes.md)) surfaced that the skills disagree on two things the `skill-help.ts` renderer had to be made tolerant of, and which the `ki-skills` rubric does not yet enforce. **Mode verbs drift:** the canonical scaffold mode is `INIT`, but `ki-decision-records`, `ki-kb-activities`, and `ki-kb-live-artifacts` use `NEW`, and `ki-feature-definitions` carries _both_ — decide whether `NEW` is an alias to retire in favour of `INIT` or a distinct mode worth defining, and settle the accepted skill-specific extensions (`OPTIMISE`, the `ki-kb-streams` lifecycle verbs) against the universal set. **Mode-heading structure drifts:** most skills use `## Mode X — gloss` (H2), `ki-authoring` uses `### Mode X` (H3), and `ki-kb` / `ki-housekeeping` declare their modes in prose with no per-mode heading at all — pick one convention so a mode list is mechanically extractable the same way everywhere (rather than relying on the renderer's argument-hint fallback). Land both as `ki-skills` rubric criteria with a mechanical check in `lint-skills.ts`, then conform the set.

### Cover the CLAUDE.md → skill move as a tokenomics lever

The [tuning guide](../../../docs/guides/user-guide/tuning.md) documents three ways to lighten the prose layer — trim `CLAUDE.md`, apply the layer scope rule (repo-specific → repo `CLAUDE.md`, cross-project → `~/.claude/*.md`, facts → memory), and reconcile a promoted memory by deleting it — but not the `CLAUDE.md` → **skill** move. `CLAUDE.md` is always-on standing surface; a skill's _body_ loads **on demand**, so only its ~50-token `description` stays in the prefix. A chunk of `CLAUDE.md` that is only _situationally_ relevant (an occasional procedure, a domain playbook) is therefore cheaper as an on-demand skill: pay the description every turn, defer the body until a task triggers it. Add a short paragraph to the guide's trim section drawing this always-on-vs-on-demand trade-off, with the caveat that only genuinely situational, self-contained content should move (lifting a one-liner costs more in standing description than it saves), and route the "is this skill-worthy" call to `/ki-skills` judgment rather than prescribing it. Consider whether it also earns a `ki-tokenomics` rubric point (progressive disclosure of the prose layer) alongside the guide prose.

### Formalize `docs/guides/agents/` as the neutral staging area for repository agent guidance _(candidate)_

Found 2026-07-15 in the `chezmoi` dotfiles repo: runtime-neutral, repository-specific agent procedures had accumulated under `.claude/*.md`, even though Codex and other agents need the same material. The repo moved those guides to `docs/guides/agents/`: `AGENTS.md` stays a compact, literal orientation and links to the detail; `CLAUDE.md` may `@`-import selected guides for Claude Code's automatic context; `.claude/` keeps only Claude runtime surfaces such as settings, commands, and linked skills. The separate `.ki-meta/` directory remains bootstrap-owned vendored tooling, never a home for authored guidance.

Decide whether this is the standard repo layout (candidate owner: `ki-agents` or `ki-repo`), then document and audit the boundaries: shared detail goes to `docs/guides/agents/`; runtime-only configuration stays in the relevant runtime directory; reusable guidance graduates into a skill rather than expanding the repository guide indefinitely. Ensure the rule composes with tokenomics: only selectively `@`-import a guide into `CLAUDE.md`; otherwise retain it as an on-demand link from `AGENTS.md`.

### Codify the convention-placement model and knowledge-promotion loop _(candidate)_

The harness governs plenty of individual placement decisions but has never stated the **model** they are all instances of: a taxonomy of _where each kind of convention or knowledge lives_, and the single principle behind it — **minimise variation that is not captured in shared, portable knowledge**; push knowledge out of machine-local and vendor-specific state so it accumulates once and transfers, rather than being re-derived per machine, per runtime, or per session. (The half-joke framing that makes it stick: shared portable knowledge is the agent estate's _non-local_ layer — it survives any single host or runtime — where machine-local state dies with the box it sits on.) Today the pieces are scattered and each was decided ad hoc: **user-specific** conventions (tone/voice, aliases, personal agents and hooks) live user-level in `~/.claude`, chezmoi-synced device-to-device (decision at "Declared per-repo agent-vendor scope" below, 2026-07-15); **project-specific** conventions live in `CLAUDE.md`/`AGENTS.md` and `docs/guides/agents/`, with `.claude/` holding only runtime surfaces (the `docs/guides/agents/` item just above); **cross-user, cross-project** standards live as harness skills, vendored into `.ki-meta/` and linked into both `~/.claude` and `.claude` (the "Vendoring architecture" theme). The routing rule between them is written twice — `~/.claude/memory-scope.md` (memory vs `CLAUDE.md`) and the tuning guide's `CLAUDE.md` → skill move ("Cover the CLAUDE.md → skill move as a tokenomics lever" above) — but neither states the overarching model, so each new placement question gets re-litigated from scratch.

The placement model names the **rungs**; it needs a companion **knowledge-promotion loop** so useful learning does not stay trapped in session state, auto-memory, or a vendor-local store. During work and at recap/improvement points, the loop should surface corrections, feedback, decisions, repeated workarounds, and emergent conventions; decide whether each is a one-off, user preference, project convention, KB knowledge, or cross-project standard; promote it to that durable home; and reconcile the lower layer by deleting the promoted memory or leaving only a pointer. A project-level rule therefore moves from auto-memory into a concise `AGENTS.md`/`CLAUDE.md` anchor plus an on-demand `docs/guides/agents/` guide; a pattern demonstrated across projects moves through the KB's governed proposal path and eventually into a canonical harness skill. Promotion is deliberate rather than automatic: recurrence, scope, and the right owning artifact require judgment. `ki-kb`'s IMPROVE mode already supplies the base-side part of this loop (memory → convention, cross-base pattern → skill REFRESH); this item generalises it across the agent estate and makes the triggers, routing, reconciliation, and escalation criteria explicit.

Land a single artifact (candidate home: a `docs/guides/` page owned by `ki-agents` or `ki-repo`, or a short standard section a REFRESH pass can check against) that names the layers, states the minimise-local-variation principle, and gives the decision procedure for placing and promoting a new convention. It must be **runtime-neutral**: name the Codex/ChatGPT equivalents of each Claude Code location (`.agents/skills`, `~/.codex/`, `AGENTS.md` — see the [runtime parity scorecard](../../../docs/decisions/references/runtime-parity-scorecard.md)) so the model reads as "where knowledge lives across the agent estate", not "where Claude Code files go". Two natural pairings: the model is the _why_ behind the multi-machine "move state out of machine-local into shared state" item (under _Multi-machine portability & shared state_ above) — that item is this principle applied to session/plan state; and it should mark, per layer, which content is **KI-opinionated** versus **genuinely portable** (the "Separate KI-opinionated conventions from genuinely general/portable standards" item under _KI philosophy vs. portable standards_ below), so the eventual extraction of a KI-neutral core has a placement map to work from. Scope the sorting pass but don't front-run those two items — this one owns the _model and its statement_; they own the state-migration and the KI/portable split respectively.

### Delineate the meta-skill ownership boundaries across the environment family _(candidate)_

Surfaced 2026-07-15 (chezmoi repo) while routing a Headroom stats-reset learning to its owning skill: the environment/meta skills that govern _the agent estate itself_ — `ki-housekeeping`, `ki-tokenomics`, `ki-skills`, `ki-authoring` — repeatedly brush against each other around memory, the standing context surface, and tool state (Headroom), and none of their descriptions states where its scope stops and the next begins, so the boundary gets re-litigated per session. The clean split, once drawn: **placement decision** (which rung a fact belongs on) is `memory-scope.md`'s tier rule plus `ki-skills`' skill-worthiness judgment; **cost pressure** pushing always-on prose down to on-demand skills is `ki-tokenomics` (it owns the per-turn standing-surface budget and the always-on-vs-on-demand call, _not_ prose style); **memory-store hygiene** — `memory/*.md` + `MEMORY.md` format, index agreement, and promote-then-delete _enforcement_ — is `ki-housekeeping` (the janitor that catches the stale leftover, not the librarian that picks the shelf); **destination format/quality** is `ki-authoring` (MD/TOML house style) and `ki-skills` (`SKILL.md` quality). This is the _who-governs-what_ axis, complementary to the convention-placement-model item above, which owns the _where-it-lives_ axis: that item names the rungs, this one assigns each rung's governance to an owning skill. The two live Headroom items under _Session tooling, memory & housekeeping_ are the worked instances — the stats-reset recipe is `ki-tokenomics` (tool operation), the `headroom:learn` cross-repo leak is `ki-housekeeping` (it dirties the memory store it governs), and the reconciliation that deletes the staged memory once either lands is `ki-housekeeping` enforcing `ki-tokenomics`'/`ki-skills`' placement call.

Land the delineation where a REFRESH pass can check it (candidate: a short "scope & off-ramps" contract each meta-skill's `SKILL.md` declares, plus a `ki-skills` rubric point requiring every skill to name its boundary and off-ramp to its neighbours), then conform the affected descriptions so `ki-housekeeping`/`ki-tokenomics`/`ki-skills`/`ki-authoring` each state their edge explicitly rather than leaving it implicit. Keep it runtime-neutral — these are governance-concern boundaries, not Claude Code file locations — and don't front-run the placement-model item, which owns the taxonomy this assigns ownership over.

### Extract the git write-safety rule into its own skill _(candidate)_

Found 2026-07-14 in the `chezmoi` dotfiles repo while reviewing whether its `.claude/*.md` files should become skills: the "don't leave `.git/*.lock` files behind, don't parallelise write-mode git commands" rule is generic agent behavior with no chezmoi/dotfiles content, yet it was duplicated in two wrong places there — a Claude Code-only, repo-scoped `.claude/git-hygiene.md`, and, worse, buried in `ki-dotfiles-chezmoi`'s `references/dotfiles-standard.md` (`## Git & audit hygiene` section), a skill whose actual scope is chezmoi dotfiles patterns. Per that repo's three-tier guidance model (project+Claude-only / project+both-agents / all-projects+all-agents), this rule belongs in the last tier as its own small skill here (candidate home: `skills/environment/`, alongside `ki-binding`/`ki-housekeeping`/`ki-tokenomics`, since it's a cross-cutting technical convention rather than a driven multi-step process). Needs proper skill scaffolding (rubric-compliant `SKILL.md`, likely via this repo's own `ki-skills`/`ki-bootstrap` tooling rather than hand-written). Once it exists: remove the duplicated line from `ki-dotfiles-chezmoi`'s `dotfiles-standard.md` (replace with an off-ramp pointer), and the `chezmoi` repo can reconsider whether its `.claude/git-hygiene.md` becomes a thin skill-pointer or is removed outright now that the ambient safety rule is skill-governed.

### Sweep decisions, feature definitions, and guides for drift and stale citations _(candidate)_

Each governance skill's own REFRESH mode checks its **standard** against its sources, and the scheduled `ki-skills-refresh` sweep honours that cadence — but nothing checks the **content written under those standards** for staleness or cross-artifact drift: whether a DR's `## References` still point at sibling records that exist and haven't been retitled, whether a feature definition's cited decisions still hold, whether a guide in `docs/guides/` still describes what the skills actually do today. `ki-decision-records`'s own mechanical checker (`audit-drs.ts`) already validates DR structure and index completeness within `docs/decisions/`, but stops at that directory's boundary — it doesn't reach into `docs/features/` or `docs/guides/` to check the links and claims that cross between them. Investigate a periodic sweep — mechanical where citation existence can be checked by a script (dead internal links, missing index entries), judgment-based where content currency is the question (does this guide's description still match the skill it documents) — scoped to run less often than per-commit, more like the `ki-skills-refresh` cadence. Decide ownership: a natural extension of `ki-decision-records`' own REFRESH, a `ki-feature-definitions` counterpart, or a cross-cutting sweep skill that walks all three surfaces in one pass.

### Roll out the expanded Feature Definitions pattern to the fleet _(candidate)_

The harness's `docs/features/` now covers the five-part bundle, the universal modes, and the checker contract as a checkable RFC-2119 inventory — a worked exemplar of `ki-feature-definitions`. Roll the same pattern out to the sibling repos (the `mcp-*` servers, `ki-website`, the KB), each getting a small `docs/features/` describing what that repo does and checkable against its own behaviour, using the harness set as the template.

### Separate KI-opinionated conventions from genuinely general/portable standards _(candidate)_

Some of what the harness currently codifies as a shared house standard is actually **KI's own philosophy or house taste** rather than a general, vendor-neutral convention any repo would want regardless of who governs it — the two are currently mixed together in the same skills/standards without a marked boundary. Once the harness itself is stable (the vendoring architecture, the cluster-subfolder layout, and the coverage cascade have settled — see "Vendoring architecture & runtime portability" above), revisit the standards corpus and identify which criteria are KI-specific opinion versus which are portable best practice, so the KI-specific pieces can be migrated to a clearly KI-scoped layer (or a KI-specific skill/table) rather than sitting undistinguished inside a standard that reads as universal. No specific criteria are called out yet — this is a placeholder to do the sorting pass, not a decision about which items move.

## Context

`ki-project-roadmap` governs the forward-work model for non-KB repositories, not only conventional projects. Rename it to `ki-repo-roadmap` so its public identity describes that scope accurately. The target is one current-state contract: all harness-owned active footprints must conform to the new name, with no alias, fallback, compatibility table, or dual-path support for the old name.

## Current state

The canonical source skill directory, `.ki-config.toml` declaration, package scripts, process-skill composition, checkers, tests, evaluation scenarios, generated graphs, documentation, and vendored local copies currently use `ki-project-roadmap`. The existing roadmap item deliberately keeps that name authoritative until this plan lands, so no opportunistic partial rename has occurred.

## Steps

1. Inventory every live harness-owned identifier and generated surface for `ki-project-roadmap`, separating canonical source from copies regenerated by bootstrap; lock `ki-repo-roadmap` as the sole target identifier and update the governing decision and feature language to state the current-state migration policy.
2. Rename the canonical skill directory and its frontmatter, then update its checker, tests, fixtures, help text, source references, configuration table, package commands, evaluation scenarios, graph edges, and direct composing consumers (`ki-plan`, `ki-next`, `ki-recap`, `ki-handoffs`, and repo-structure skills).
3. Update current user, developer, feature, decision, and generated-graph documentation to use `ki-repo-roadmap`; preserve only unavoidable historical prose in closed records, never an executable or supported old-name contract.
4. Re-run the repository bootstrap and generated-surface commands so `.ki-meta`, `.claude`, `.agents`, command launchers, skills-map output, roadmap index, and other vendored copies are regenerated from the renamed canonical source rather than hand-edited.
5. Sweep the source, configuration, generated copies, and command surface for stale live `ki-project-roadmap` references; correct every active footprint and document a direct conform/reinstall route for external repositories rather than shipping compatibility logic.
6. Run focused renamed-skill tests and the full harness gates, then mark each completed step in this plan and commit the verified migration in coherent units.

## Files touched

- `skills/general-governance/ki-project-roadmap/` → `skills/general-governance/ki-repo-roadmap/`, including scripts, references, and tests.
- `.ki-config.toml`, `package.json`, `evals/`, `docs/`, and all first-party skills that name or compose the roadmap standard.
- Generated bootstrap output under `.ki-meta/`, `.claude/`, `.agents/`, and the generated skills-map and roadmap index artifacts.

## Verify

1. `bun skills/general-governance/ki-repo-roadmap/scripts/project-roadmap.test.ts` passes.
2. `bun run ki:skills:graph`, `bun run test`, and `bun run ki:audit` pass serially.
3. A scoped repository sweep finds no live source, configuration, checker, command, or generated-copy use of `ki-project-roadmap`; the only permitted remaining occurrences are historical closed-record context and this retiring plan/roadmap item before closure.
4. A fresh bootstrap of this harness produces only the `ki-repo-roadmap` checker and commands.

## Dependencies / blocks

GOV-003 blocks this plan because it changes the shared skill frontmatter and generated dependency graph that this rename must migrate only once. This plan then blocks GOV-002, whose structure-aware process design must compose the final `ki-repo-roadmap` identity rather than introduce another rename surface.

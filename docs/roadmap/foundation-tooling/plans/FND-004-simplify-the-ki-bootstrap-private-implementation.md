## Context

The first structural split now separates one-time user installation from repository bootstrap under `scripts/internal/user-install/` and `scripts/internal/repo-bootstrap/`. That makes the two public responsibilities visible, but the repository-bootstrap implementation remains difficult to reason about and difficult to reuse safely. This plan establishes clear internal ownership before adding more lifecycle behaviour.

## Current state

The user-install subtree is a bounded four-file implementation. The repository-bootstrap subtree contains resolution, skill graph/help generation, shared-module synchronisation, `.ki-meta/` generation, repository-local educator generation, runtime skill/agent publication, and two separate fail-safe publication mechanisms. Its principal engine is about 1,875 lines and its runtime publisher about 755 lines. Skill-tree discovery is duplicated between the resolver and publisher, `runtime-paths.ts` contains at least one apparently unused writer, and some generated source templates sit inside the main orchestrator. The transaction protecting `.ki-meta/` and the transaction protecting runtime payloads have different publication boundaries and must not be merged merely because both use staging and rollback.

## Steps

1. ✓ Inventory every internal file, public caller, import edge, generated output, and mutation boundary. Classify each unit as user installation, repository orchestration, resolution/metadata, generated payload construction, `.ki-meta/` publication, runtime publication, graph/help tooling, or shared-module synchronisation. Record which responsibilities belong to `ki-bootstrap` and which are candidates for another owning skill; make no moves during this analysis pass.
2. ✓ Define and review the target module map. Keep `repo-bootstrap.ts` as a thin CLI/orchestrator, retain separate safety boundaries for `.ki-meta/` and runtime payload publication, choose one canonical skill index/resolution implementation, and identify generated templates or pure helpers that can move without expanding the dependency surface. Prefer a few coherent modules over fragmented one-function files.
3. Remove only evidence-backed dead code and duplication. Start with unused exports such as `ensureGitignore` if call-site and test searches confirm they are dead, then consolidate duplicated skill discovery behind the canonical resolver. Do not retain compatibility aliases or old internal paths.
4. Extract the minimum self-contained repository-bootstrap capability needed by skill EDUCATE entrypoints into `scripts/shared/` and declare it through `ki-shared-modules:`. `ki-bootstrap` is the provider and first user: it invokes the canonical module from `scripts/shared/`, while other skills declare `ki-bootstrap:<module>` through `ki-shared-dependencies:` and invoke their local `scripts/vendored/ki-bootstrap/` copy. The shared closure must not import private `scripts/internal/` code or cross a skill boundary at runtime.
5. Refactor the remaining repository-bootstrap implementation along the reviewed module map while preserving CLI behaviour, dry-run/verbose semantics, generated bytes where contractual, and all symlink, identity, race, rollback, ownership, and mode protections. Keep user-install internals separate and unchanged unless the inventory proves a shared primitive has the same contract in both domains.
6. Update source callers, package scripts, tests, guidance, and the `ki-bootstrap` rubric to the final internal boundaries. Re-vendor the governed footprint once after source changes settle; do not create transitional paths.
7. Prove behaviour with focused resolver, shared-module, repository-bootstrap parity/safety, runtime-publication, temporary-source-removal, and local EDUCATE tests, followed by the serial repository gates.

## Step 1 inventory

This is an analysis-only snapshot of the implementation before any internal moves. Paths in the first two sections are relative to `skills/keystone/ki-bootstrap/scripts/internal/`.

### User installation

- `user-install/user-install.ts` is the user-install orchestrator and publisher. It detects or accepts Claude Code and Codex runtimes, copies the five global process/bootstrap skills, maintains the generated-install marker, and delegates Claude hook installation. Its mutation boundary is the selected user runtime skill directories.
- `user-install/install-claude-hook-payload.ts` is the Claude hook payload publisher. It owns the locked, content-addressed hook namespace and stable active command copies below the user's Claude directory; it does not write Claude settings.
- `user-install/user-install.test.ts` covers runtime detection, replacement, dry-run/check behaviour, managed-path safety, and the hook-install handoff.
- `user-install/install-claude-hook-payload.test.ts` covers hook payload integrity, upgrades, locking, identity, rollback, symlink safety, and the public shell entrypoint.

These four files form a bounded user-space responsibility and remain owned by `ki-bootstrap`. They share no private imports with the repository-bootstrap subtree; no common primitive is yet evidenced strongly enough to join the two domains.

### Repository bootstrap

- `repo-bootstrap/repo-bootstrap.ts` currently combines repository orchestration, generated payload construction, `.ki-meta/` candidate construction, and fail-safe `.ki-meta/` publication. It also delegates config scaffolding and runtime publication.
- `repo-bootstrap/resolve.ts` owns the source skill index, config and frontmatter interpretation, dependency validation, selected-set resolution, vendor-unit discovery, and shared-module payload resolution.
- `repo-bootstrap/runtime-paths.ts` supplies runtime path and `.gitignore` helpers. It is byte-identical to the private `ki-repo` copy, and its exported `ensureGitignore` writer has no call site.
- `repo-bootstrap/project-skill-publisher.ts` inspects and publishes regular-file runtime skills plus Claude governance-agent links. It owns a separate repository-root staging, quarantine, rollback, and `.gitignore` transaction from `.ki-meta/` publication.
- `repo-bootstrap/publish-project-skills.ts` is the thin public CLI adapter over the runtime publisher.
- `repo-bootstrap/sync-shared-modules.ts` validates or replaces source-tree `scripts/vendored/<provider>/` payloads from declared shared dependencies.
- `repo-bootstrap/skill-graph.ts` validates and renders the source skill dependency graph; it is also copied to `.ki-meta/bin/` for harness-shaped targets.
- `repo-bootstrap/skill-help.ts` renders source skill HELP and catalogue coverage; it is also copied to `.ki-meta/bin/` for harness-shaped targets.
- `repo-bootstrap/resolve.test.ts` mixes focused resolver coverage with bootstrap, aggregate-runner, vendoring, temporary-source, and bootstrap-audit integration coverage.
- `repo-bootstrap/repo-bootstrap.safety.test.ts` covers the `.ki-meta/` transaction's path, symlink, identity, race, rollback, ownership, and mode protections.
- `repo-bootstrap/repo-bootstrap.parity.test.ts` covers generated checker and wrapper parity across direct, package, and vendored entrypoints.
- `repo-bootstrap/skill-graph.test.ts` covers graph validation, documentation projection, CLI errors, and generated-byte parity.
- `repo-bootstrap/sync-shared-modules.test.ts` is the source shared-module drift gate.

The repository-bootstrap subtree has four distinct write boundaries that must stay explicit: `ki-repo` owns `.ki-config.toml` scaffolding invoked by subprocess; the bootstrap transaction owns `.ki-meta/`; the runtime publisher separately owns generated runtime skills, Claude agent links, and `.gitignore`; and shared-module synchronisation mutates source `scripts/vendored/` payloads.

### Imports and callers

The private import graph is small despite the large files:

- `repo-bootstrap.ts` imports `resolve.ts`.
- `resolve.ts` imports `runtime-paths.ts`.
- `project-skill-publisher.ts` imports `resolve.ts` and `runtime-paths.ts`.
- `publish-project-skills.ts` imports `project-skill-publisher.ts`.
- `sync-shared-modules.ts` imports `resolve.ts`.

The public and cross-boundary callers are broader:

- `repo-bootstrap.sh`, the package `ki-bootstrap` bin, and `ki-bootstrap/scripts/educate.ts` call the private repository engine directly.
- `user-install.sh` calls the private user installer, which invokes the hook-payload installer for Claude Code.
- `repo-bootstrap.ts` invokes `ki-repo` config scaffolding, `skill-help.ts`, and `publish-project-skills.ts` by subprocess.
- `ki-bootstrap` rubric contexts import the private resolver and runtime publisher, and invoke the publisher for conform actions.
- `package.json` and the `ki-repo` development-link tests call `publish-project-skills.ts`; package tests also call `skill-graph.ts` and `skill-help.ts` directly.
- `ki-repo/scripts/educate.ts` and `ki-skills/scripts/educate.ts` still call bootstrap source across a skill boundary. Twenty-two other skill EDUCATE entrypoints point at the intended but absent `ki-bootstrap/scripts/lib/repo-bootstrap.ts` surface. Step 4 replaces both shapes with declared local shared-module consumption.

### Generated outputs

Repository bootstrap constructs `.ki-meta/checkers/<skill>/`, `.ki-meta/educators/<skill>/`, `.ki-meta/bin/{aggregate.ts,ki-audit,ki-conform,ki-educate,ki-help}`, and `.ki-meta/manifest.json`. Harness-shaped targets additionally receive `skill-graph.ts` and `skill-help.ts` in `.ki-meta/bin/`. The aggregate runner, shell launchers, educator launcher, command wrapper, and manifest are currently embedded templates inside `repo-bootstrap.ts`.

Runtime publication separately creates generated regular-file skills under each declared runtime, Claude governance-agent links, generated-skill markers below each copied skill's `.ki-meta/`, and matching `.gitignore` entries. User installation creates managed global skill copies and the durable Claude hook payload. Shared-module synchronisation creates source-local vendored module copies.

### Ownership conclusions

`ki-bootstrap` owns user installation, repository-bootstrap orchestration, target coverage resolution, generated payload construction, `.ki-meta/` publication, and normal runtime publication. The `.ki-meta/` and runtime transactions protect different targets and remain separate.

The following are ownership or extraction candidates for later reviewed steps, not moves approved by this inventory:

- The source dependency graph and HELP renderer describe the skill system and are candidates for `ki-skills` ownership; bootstrap would remain their publisher.
- The aggregate checker runner's schema and reporting behaviour are candidates for the canonical `ki-skills` checker/rubric modules; bootstrap would remain responsible for assembling and publishing the target payload.
- Source shared-module synchronisation enforces skill packaging and is a candidate for `ki-skills` ownership, although repository bootstrap still needs shared-module resolution and copying while it builds `.ki-meta/`.
- Runtime path helpers and the runtime publisher should have one canonical implementation. The current `ki-repo` private copies are downstream duplication; `ki-repo` should consume an explicit local shared module for its development-link capability rather than import bootstrap internals.
- Skill-index discovery is repeated in the resolver, runtime publisher, user installer, graph, and HELP tools. Consolidation must respect their different deployment closures rather than create a private cross-skill dependency.
- `ensureGitignore` is an evidence-backed dead-code candidate, while the mixed `resolve.test.ts` responsibilities are a later test-boundary cleanup candidate.

## Step 2 target module map

The reviewed target keeps the public shell entrypoints unchanged and uses a small number of responsibility-sized TypeScript modules.

### Private repository engine

- `scripts/internal/repo-bootstrap/repo-bootstrap.ts` remains the CLI and workflow orchestrator. It parses arguments, binds the target identity, resolves the harness ref and selected skill set, composes `ki-repo` config scaffolding, invokes the two publishers in order, and owns only top-level progress and completion reporting.
- `scripts/internal/repo-bootstrap/resolve.ts` remains the canonical source index and metadata resolver for repository bootstrap, shared-module resolution, and runtime publication. The runtime publisher stops maintaining its own skill index. Graph, HELP, and user installation retain their small self-contained discovery closures because they are deployed independently.
- `scripts/internal/repo-bootstrap/generated-payloads.ts` becomes the pure source-builder module for the aggregate runner, executable bin wrappers, generated educator launcher, and command wrapper. It performs no filesystem access or publication.
- `scripts/internal/repo-bootstrap/meta-publisher.ts` owns construction and fail-safe publication of one complete `.ki-meta/` generation. Candidate building, entry snapshots, the lock, staging, quarantine, publication, rollback, and manifest-last rule stay together because they share one safety invariant; they are not abstracted into a generic transaction framework.
- `scripts/internal/repo-bootstrap/project-skill-publisher.ts` remains the independent runtime publication transaction for generated runtime skills, Claude governance-agent links, markers, and `.gitignore`. It exposes direct inspect and publish functions for the orchestrator, CLI adapter, and rubric contexts instead of requiring internal callers to reconstruct command-line arguments.
- `scripts/internal/repo-bootstrap/publish-project-skills.ts` remains the thin standalone CLI adapter over the runtime publisher.
- `scripts/internal/repo-bootstrap/runtime-paths.ts` remains the shared private runtime/config path helper used by resolution and runtime publication. The unused writer is removed. Retiring the duplicated `ki-repo` publication closure requires a separately declared shared-module boundary and stays outside FND-004 rather than creating a private cross-skill import.

The internal dependency direction is one-way: the orchestrator calls resolution and the two publishers; the `.ki-meta` publisher calls resolution and pure payload builders; the runtime publisher calls resolution and runtime-path helpers. The publishers do not import each other, and no module imports the orchestrator.

### Portable EDUCATE transport

`scripts/shared/repo-bootstrap.ts` is a generic, self-contained transport client, not a copy or facade of the private repository engine. It accepts the target plus repeatable `--seed`, `--ref`, `--dry-run`, and `--verbose` arguments and invokes the public repository-bootstrap transport. It uses only platform/Bun facilities and imports nothing from `scripts/internal/`.

`ki-bootstrap` declares that file as a shared module and uses the canonical copy from `scripts/shared/`. Every other governance skill that delegates EDUCATE declares `ki-bootstrap:repo-bootstrap`, receives it below `scripts/vendored/ki-bootstrap/`, and invokes only that local copy with its own `--seed`. Source skills, installed skills, and vendored skills therefore have the same boundary without reaching across skill directories.

The generated target-local educator launchers remain payloads produced by `generated-payloads.ts`; they are not runtime imports of the source shared module. This preserves the standalone `.ki-meta/educators/<skill>/educate.ts` contract after temporary bootstrap source is removed.

### Utilities held stable

- `skill-graph.ts` and `skill-help.ts` remain standalone, single-file tools during FND-004 so their harness-only `.ki-meta/bin/` copies stay self-contained. A separate ownership decision may move their canonical source to `ki-skills`; this refactor does not couple them to `resolve.ts`.
- `sync-shared-modules.ts` remains a private source-maintenance CLI during FND-004. Repository bootstrap continues resolving and copying declared shared payloads independently while assembling `.ki-meta/`.
- The user-install subtree remains unchanged and does not import repository-engine modules.

### Refactoring constraints

- Preserve `.ki-meta/` and runtime publication as separate transactions with separate staging, guards, rollback, and tests; do not create a generic transaction abstraction between them.
- Keep safety helpers private to the publisher whose invariant they enforce. Extract a helper only when it removes real duplication or names a meaningful operation.
- Preserve public CLI arguments, dry-run and verbose behaviour, generated bytes where contractual, and manifest-last publication.
- Do not add compatibility paths for the current private layout. Update current callers and vendored consumers to the reviewed boundaries in one migration.
- Keep graph/HELP ownership, shared-module tooling ownership, checker-runner ownership, and user-install changes outside this structural pass unless a later step obtains a separate explicit decision.

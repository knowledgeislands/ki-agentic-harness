### Replace `ki-plans` with `ki-project-roadmap`

Recast non-KB work governance around the artifact that leads it: replace the `ki-plans` governance skill with `ki-project-roadmap`, while retaining `ki-plan` as the process skill that creates, promotes, executes, and closes individual plans. Support two progressively disclosed project-repo profiles without a config switch: **simple** uses only the root `ROADMAP.md`; **thematic** keeps that root as a compact generated portfolio projection and moves canonical detail into `docs/roadmap/<theme>/ROADMAP.md`, with active execution plans under `docs/roadmap/<theme>/plans/<NNN>-<slug>.md` and a global index/dependency graph under `docs/roadmap/README.md`. A simple item expands into a theme when its detail or execution needs outgrow the root; each item has one authoritative home, and plans use a qualified theme/item locator rather than ambiguous title-only linkage. The standard, checker, CONFORM, and `ki-plan` lifecycle must preserve the five horizons, near-horizon plan rule, global plan ids, no-clobber promotion transaction, and open-only closure across both profiles. Knowledge Base repos are an explicit off-ramp: `ki-kb-streams` replaces this standard entirely because streams are already thematic roadmaps, proposals are roadmap items, and proposal checklists are their plans; never create `ROADMAP.md` or `docs/roadmap/` in a KB. Ratify the boundary and migration in a Decision Record, then update the canonical skill name, config marker, implication/dependency graph, scripts, vendored units, documentation, active artifacts, and downstream fleet references directly to the target contract, without a legacy compatibility bridge.

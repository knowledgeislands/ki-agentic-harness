### Hooks as a governed harness surface

**Sequence.** Framing → mechanism → integrity → ergonomics. The "vendor-in, then use in-repo" framing (and the AI-multiplier framing) is the _why_ the rest answer to — state it first. Whole-skill vendoring into `.ki-meta/skills/` is the mechanism; the source-vs-vendored hash check extends naturally once it lands, and hooks-as-governed-surface + per-repo agent-vendor-scope are the conditional-vendoring cases of the same mechanism. The thin `ki` CLI is a pure wrapper over the result — build it last, after the shell-wrapper-vs-`package.json` parity test under _Next_ has settled the invocation paths it must match.

**Shipped (2026-07-12):** the motivating plan-file lifecycle pair — `hooks/plan-stamp.sh` (`PostToolUse(ExitPlanMode)`, stamps `status`/`created`/`cwd` frontmatter) and `hooks/plan-sync.sh` (`PostToolUse(TodoWrite)`, mirrors live todo state into the plan and flips `status: open → done` on completion) — landed **directly as a harness-owned surface**, not staged through chezmoi first as originally sketched here. `hooks/` graduated from "Shelf" to "Populated (partial)"; install is `bun run ki:hooks:link:global` (`skills/keystone/ki-bootstrap/scripts/link-hooks.ts`, cloned from `skills/keystone/ki-bootstrap/scripts/sync-skills.ts`'s global-link mode), which symlinks the scripts into `~/.claude/hooks/` and merge-patches `~/.claude/settings.json`'s `hooks.PostToolUse` directly. Hardened against path-traversal, symlink-follow writes/reads, and malformed-JSON crashes via an adversarial safety review. The personal chezmoi hooks (`chezmoi-drift.sh`, `git-lock-check.sh`, `auto-snapshot.sh`) remain unmanaged and un-audited — genuinely personal/environment concerns, not repo-portable, so out of scope for harness ownership.

**Hooks are the odd part out — Claude Code-specific where the other surfaces are not.** Skills follow the vendor-neutral [Agent Skills standard](https://agentskills.io/); MCP servers sit on an open protocol; agents and evals are Markdown/scenario artifacts whose substance is runtime-portable even where the frontmatter is Claude-flavoured. Hooks alone are executable automation bound to one runtime's proprietary surface: Claude Code's lifecycle events (`PreToolUse`/`PostToolUse`/`SessionStart`/`Stop`/`PreCompact`) and its `settings.json` registration format. The standard must own that coupling explicitly: what the harness governs today is _Claude Code_ hooks, and a second runtime's hook system, should one arrive, would be a sibling surface with its own registration model — not an extension of this one.

**Decided: a governance skill is wanted** — no longer "whether", only shape and name. It AUDITs/CONFORMs registered hooks the way the other surfaces are governed: house script style (`set -u`, exit-0-always, `jq`-parsed stdin, atomic `mktemp`+`mv` writes), safety posture (resolved-path containment, symlink refusal, unterminated-marker fail-safe — the adversarial-review findings from the shipped pair are the seed rubric), linker/`settings.json` registration integrity, and the user-level installation documented in `hooks/README.md`. Naming is the open sub-question: `ki-hooks` matches the surface-name pattern (`ki-skills`, `ki-agents`, `ki-mcp`); `ki-claude-hooks` makes the vendor coupling explicit in the name, matching the reality that the standard's content is Claude Code-specific. Cross-cuts the `ki-recap` process skill, which already names "a hook" as a destination for harvested learnings.

**Decided 2026-07-15 — graduate `git-lock-check.sh` in, reversing the note above.** The chezmoi repo's `git-lock-check.sh` `Stop` hook (clears stale `.git/*.lock` files left by a killed command) is directly comparable to the shipped `plan-stamp.sh`/`plan-sync.sh` pair — a user-level, machine-wide hook, not a repo convention — so it graduates to harness ownership under this same surface rather than staying a personal chezmoi hook (leaving `auto-snapshot.sh`, and the already-dropped `chezmoi-drift.sh`, as the remaining personal ones). Ride it in with the current plan-hook work: add `hooks/git-lock-check.sh` + `hooks/git-lock-check.test.ts`, document the third hook in `hooks/README.md` (its one difference is event `Stop`/matcher `*`, not the plan pair's `PostToolUse`), and generalize `skills/keystone/ki-bootstrap/scripts/link-hooks.ts` (add to `HOOK_NAMES`; give `HookPair` an `event` field; the `cmdPatchSettings`/`cmdCheck` paths are currently hardcoded to `hooks.PostToolUse` and must key each pair off its own event). Its _governance_ still awaits the skill decided just above; its _conventions_ counterpart is the separate git-write-safety-skill item under **Soon**. The chezmoi side (delete the source, `.chezmoiignore`/`.chezmoiremove`, `CLAUDE-CODE-SETUP.md` fix) is a dependent follow-on tracked in that repo's own ROADMAP.

### Declared per-repo agent-vendor scope, driving conditional vendoring _(candidate)_

The harness itself stays runtime-portable ([SDR-KI-HARNESS-002](../../../docs/decisions/SDR-KI-HARNESS-002-runtime-portable-contracts.md): Claude Code today, Hermes/Pi/Codex tomorrow), but a given _repo_ installing from it may only ever run one vendor's agent — there is currently no way for a repo to say so. Add a declared key (candidate home: `.ki-config.toml`'s `[ki-repo]` table, or a new `[ki-bootstrap]`/`[ki-tokenomics]` table if the axis belongs closer to install-time than repo identity) listing the agent vendor(s)/runtimes a repo works with — e.g. `["claude-code"]`, `["claude-code", "codex"]`, or `"any"` for vendor-agnostic. Once declared, this becomes the signal that drives **conditional vendoring** of runtime-bound surfaces: the hooks surface just above is Claude Code-only precisely because nothing today lets a repo opt out of installing it, or lets a future Hermes/Pi-specific hook set install only where declared. Same shape as `ADR-KI-HARNESS-009`'s portable model types (a portable declaration, resolved to vendor-specific concretes at install time) but for _which vendor(s) run here_ rather than _which tier to use_. Needs design before building: owning skill, exact key name/shape, and whether `ki-bootstrap`'s vendoring (`bootstrap.ts`, `link-hooks.ts`) reads it to skip vendor-mismatched artifacts.

**Decision (2026-07-15):** keep personal agents user-level in both Claude Code and Codex; hooks are user-level too, wherever a runtime supports them. The harness should not add project-local agent or hook linking merely for parity; this repo consequently leaves `[ki-agents]` absent. This item's runtime declaration is still useful for determining which user-level runtime-bound integrations can be configured, but not for moving personal automation into repositories.

### State "vendor tools into a project, then use them from there" as the harness's explicit core framing _(candidate)_

`CLAUDE.md`'s "What this repo is" describes the skill taxonomy (governance vs process, the six clusters) but never states outright what the harness is _for_: it exists to **vendor** tooling — skills, agents, checkers, and (candidate) hooks — into a consuming repo's own tree, so that repo can run and audit against them without depending on this harness's checkout at runtime. The three items just above are all instances of this one principle, not independent concerns: the `.ki-config.toml` schema formalises what gets declared before vendoring; declared per-repo agent-vendor scope decides _what_ is fit to vendor for a given repo; vendoring skills into `.ki-meta/skills` (rather than symlinking outside the repo) is the mechanism itself, generalising the pattern already proven for checkers (`bootstrap.ts`) to every other surface. Once that pattern reaches skills, agents, and hooks alike, "vendor-in, then use in-repo" is the harness's actual value proposition — worth stating explicitly in `CLAUDE.md`'s "What this repo is" (and possibly the README) rather than leaving it implicit in how `ki-bootstrap` happens to work, so a new contributor or a REFRESH pass can judge new surfaces against it directly instead of re-deriving it each time.

**Added framing (2026-07-13):** the shape is a chezmoi-link approach for a project — a fixed, versioned set of tooling linked in from one vendored source of truth — but scoped to AI-agent tooling _and_ ordinary dev tooling alike (e.g. shared `biome`/`tsconfig` config could vendor the same way skills and checkers do, not just agent-facing surfaces). Vendoring into `.ki-meta` also buys **reuse across multiple repos from one pinned copy** and **fixes a repo to a declared tooling set** rather than floating against whatever this harness's checkout happens to contain at audit time — the same reproducibility argument as vendoring dependencies, but deliberately not the same _mechanism_: this is explicitly not a `node_modules`. The boundary is size/kind, not principle — small, house-authored tooling (skills, checkers, shared configs) is in scope; pulling in large third-party codebases is not, and should stay on the existing dependency-manager path (`bun install`, etc.) rather than being folded into `.ki-meta`.

### Frame the harness as increasing the odds AI's multiplier effect lands positive _(candidate)_

AI is a multiplier on whatever it's pointed at — it doesn't have an inherent sign, it takes the sign of what feeds it: good practices, clear intent, and disciplined process get multiplied into faster, better outcomes; sloppy practices and unclear goals get multiplied into bigger messes, faster. A harness doesn't guarantee a positive outcome — it increases the _probability_ the multiplier lands positive. It does this two ways: **guardrails** (the governance skills' AUDIT/CONFORM gates, the checker contract, adversarial safety-review for auto-executing surfaces) constrain the downside tail; **tools** (skills, agents, MCP servers) are targeted upgrades — each governed surface acts like a named enchanted item with a specific bonus against a specific failure mode (e.g. `ki-decision-records`' audit as a "+2 sword of justice" against untraceable decisions), not a generic buff. Worth stating explicitly alongside the vendoring-framing item above, as the _why_ to that item's _how_.

### Vendor skills into `.ki-meta/skills`, then symlink `.claude/skills` from there — no symlink target outside the repo _(candidate)_

Today `ki:skills:link:project` (`skills/keystone/ki-bootstrap/scripts/link-skills.ts`) symlinks a consumer repo's `.claude/skills/<skill>` straight at this harness's own `skills/<cluster>/<skill>` — a relative symlink that resolves **outside the consuming repo entirely**, onto wherever this harness happens to be checked out locally. That is a different pattern from how checkers are already handled: `bootstrap.ts` vendors (copies) each declared skill's `audit.ts`/`conform.ts` into the repo's own `.ki-meta/skills/<skill>/`, precisely so the repo is self-governing without depending on an external checkout path — the same reason installed scripts must be copies, not symlinks (rubric `SCRIPT-7`). Apply the same discipline one level up: vendor the **whole skill directory** (not just its scripts) into `.ki-meta/skills/<skill>/`, then have `.claude/skills/<skill>` symlink to that in-repo vendored copy instead of reaching across the filesystem to this harness's checkout. Generalises to anything else vendored (agents, hooks): the in-repo `.ki-meta/` copy is the source of truth a symlink may point to; a symlink should never leave the repo boundary. Needs design: `.ki-meta/skills/` vs the existing `.ki-meta/<skill>/` checker layout (folded together or kept separate), how `link-skills.ts`'s `--check`/orphan/broken-link detection changes when the symlink target is local, and whether this changes the re-vendor workflow described in this repo's own `CLAUDE.md` (re-bootstrap after editing a canonical checker).

### A thin `ki` command-line tool wrapping the vendored scripts _(candidate)_

Requested 2026-07-15: give the harness a globally-installed command-line tool the way [Headroom](https://extraheadroom.com) ships one — a short `ki` command on `PATH` — rather than requiring per-repo invocation. Today a governed repo self-governs two ways, both awkward for a user at a shell: `bun run ki:audit`/`ki:conform` (only where the repo carries the `ki-engineering` toolchain and a `package.json` with those keys) or `./.ki-meta/bin/ki-audit` (works package.json-free, but the user must be at the repo root and know the vendored path). A `ki audit` / `ki conform` on `PATH` would walk up from the current directory to the nearest `.ki-meta/bin/` and `exec` its wrapper — so any bootstrapped repo audits or conforms with one short command from anywhere in its tree, regardless of whether it has a `package.json`. Keep it **super-thin**: no logic of its own, purely discovery-plus-delegation to the vendored `aggregate.ts`, exactly mirroring how `.ki-meta/bin/ki-audit` already `exec bun`s it — the tool must never reimplement or drift from the vendored checkers, only locate and hand off to them. Natural fit for the `ki-tools` standard (one standalone CLI per repo, `curl | bash` installer + Homebrew tap formula) as a new `tools-ki` repo, distributed alongside `tools-mgit`. Needs design: how it reconciles with the two existing invocation paths (pairs with the "Explicitly test the `.ki-meta/bin/ki-audit`/`ki-conform` shell wrapper against the `bun run ki:*` path" item under _Next_ — the tool becomes a third form that must produce identical exit codes and findings); whether it lives as a `tools-*` repo or ships from the harness itself; and how upward `.ki-meta/bin/` discovery behaves across nested or sibling governed repos.

### Port the KI MCP servers into Cowork's sandbox

The `ki-binding` cross-surface skill is built, and the Cowork leg ships the `knowledgeislands/ki-plugins` marketplace plugin — but **skills + agents only**. The KI MCP servers are host-local (they read host filesystem paths and resolve secrets via 1Password) and Cowork runs plugins in a gVisor sandbox that cannot reach the host, so the servers do not port as-is (verified 2026-07-06, [cross-surface-enablement.md](../../../skills/repo-structure/ki-mcp/references/cross-surface-enablement.md) Verification log). Making them reachable needs one of: a self-contained server bundled via `${CLAUDE_PLUGIN_ROOT}`, mounting the KB into the sandbox, or authenticated remote (`http`/`sse`) endpoints. Then `build-plugin.ts` emits a `.mcp.json` and the `cowork` `clients` token becomes live. Related open question: `ki-plugins` keeps the harness's proprietary LICENSE while public, which conflicts with `ki-repo`'s public-repo-MIT check — resolve the licensing/visibility stance. Separately, Cowork supports a browser-control plugin capability that Claude Code's harness currently has no equivalent for (unverified against current docs as of 2026-07-14) — worth confirming what that capability actually offers before deciding whether it changes the shape of this port.

### Run claude.ai connectors' equivalents behind mcporter _(candidate)_

Several claude.ai managed connectors have a KI-owned local equivalent already in the estate — Google Calendar/Drive/Gmail ↔ `mcp-gsuite`, Microsoft 365 ↔ `mcp-m365` — so the same capability could be served from the local server set behind mcporter rather than each surface authorising a separate hosted connector. This lands cleanly on the **local** surfaces (Claude Code, Desktop): the server sits behind the single `ki-mcporter` slot ([ADR-KI-HARNESS-TOOLCHAIN-003](../../../docs/decisions/ADR-KI-HARNESS-TOOLCHAIN-003-mcporter-mcp-proxy.md)) and Tool Search keeps its schemas off the standing surface until used. The blocker is the same host-local reachability wall as the Cowork item above: **claude.ai (web) cannot reach a localhost daemon**, so parity there needs an authenticated public endpoint (a tunnel + auth) registered as a custom connector — a deliberate security decision — or accepting that web keeps the hosted connector while local surfaces use the mcporter-fronted server. Investigate the mapping (which connectors have a local equivalent worth running, which are better left hosted — Slack/Linear/Attio/Xero/Granola have no KI server), the tunnel/auth posture, and how `ki-binding`'s `clients:` targeting expresses "local surfaces use the server, web uses the connector" without the surfaces silently diverging.

### Run Hermes remotely for background asynchronous work _(candidate)_

[Hermes](https://github.com/nousresearch/hermes-agent) is a candidate host for offloading long-running or overnight agent work — the goal is background async execution (e.g. while asleep or away from the laptop), not a replacement for the interactive harness workflow; keep Hermes a tool invoked for a specific class of work, not the main focus of daily use. Two candidate homes: the Mac Studio (always-on, local network, no extra cost) versus an AWS account (reachable from anywhere, but adds cloud spend and a security/access-control surface to manage). Evaluate: which harness surfaces (skills/agents/MCP) Hermes needs to reach and whether that repeats the host-local reachability wall already tracked above (Cowork sandbox, mcporter web-parity items) — a remote Hermes instance likely needs the same authenticated-endpoint treatment as those; how a job gets handed to Hermes and how its results get back into the harness/KB; and what "done, needs review" surfacing looks like so background runs don't silently pile up unreviewed.

# Runtime portability roadmap

## Blocking

## Next

## Soon

### Route multi-machine harness state through portable or synchronized homes

Make laptop/desktop handoff predictable by inventorying genuinely machine-local harness and session state — per-project auto-memory, user settings and hooks, Headroom learned patterns, and other runtime caches — then assigning each class one durable home: repository-tracked state, KB content, chezmoi-synchronized personal state, or intentionally local disposable state. Project roadmaps and their plans remain repository-tracked; KB repositories continue to use streams and proposal checklists rather than absorbing code-repo plans.

Produce a finite routing table or decision record, then create follow-on migration items only for state currently in the wrong home. Relate the outcome to the remote Hermes and mcporter reachability work, but do not make those implementations prerequisites for deciding where state belongs.

### Use `target_runtimes` to conditionally install runtime-bound surfaces _(candidate)_

The portable declaration has shipped: `[ki-repo] target_runtimes` accepts Claude Code and Codex targets, and the skill linkers use it to select their user-level runtime destinations. The remaining work is to decide which runtime-bound integrations consume the same signal and skip incompatible installation. Claude Code hooks are the first concrete case; future runtime-specific agents or hook sets should follow the same declared selection rather than probing the machine or installing everywhere.

Keep personal agents and hooks user-level wherever a runtime supports them; `target_runtimes` chooses which user-level integrations are configured, not whether personal automation moves into a repository. Design the ownership and installer boundary before implementation: whether `ki-bootstrap` coordinates runtime-bound linkers directly, or each governed surface reads `[ki-repo] target_runtimes` through its own linker while bootstrap only sequences them.

### State "vendor tools into a project, then use them from there" as the harness's explicit core framing _(candidate)_

`CLAUDE.md`'s "What this repo is" describes the skill taxonomy (governance vs process, the six clusters) but never states outright what the harness is _for_: it exists to **vendor** tooling — skills, agents, checkers, and runtime-bound integrations — into a consuming repo's own tree, so that repo can run and audit against them without depending on this harness's checkout at runtime. The `.ki-config.toml` schema formalises what gets declared, `target_runtimes` selects compatible runtime integrations, and whole-skill vendoring generalises the pattern already proven for checkers. Once that pattern reaches skills, agents, and hooks alike, "vendor-in, then use in-repo" is the harness's actual value proposition — worth stating explicitly in `CLAUDE.md`'s "What this repo is" (and possibly the README) rather than leaving it implicit in how `ki-bootstrap` happens to work, so a new contributor or a REFRESH pass can judge new surfaces against it directly instead of re-deriving it each time.

**Added framing (2026-07-13):** the shape is a chezmoi-link approach for a project — a fixed, versioned set of tooling linked in from one vendored source of truth — but scoped to AI-agent tooling _and_ ordinary dev tooling alike (e.g. shared `biome`/`tsconfig` config could vendor the same way skills and checkers do, not just agent-facing surfaces). Vendoring into `.ki-meta` also buys **reuse across multiple repos from one pinned copy** and **fixes a repo to a declared tooling set** rather than floating against whatever this harness's checkout happens to contain at audit time — the same reproducibility argument as vendoring dependencies, but deliberately not the same _mechanism_: this is explicitly not a `node_modules`. The boundary is size/kind, not principle — small, house-authored tooling (skills, checkers, shared configs) is in scope; pulling in large third-party codebases is not, and should stay on the existing dependency-manager path (`bun install`, etc.) rather than being folded into `.ki-meta`.

### Frame the harness as increasing the odds AI's multiplier effect lands positive _(candidate)_

AI is a multiplier on whatever it's pointed at — it doesn't have an inherent sign, it takes the sign of what feeds it: good practices, clear intent, and disciplined process get multiplied into faster, better outcomes; sloppy practices and unclear goals get multiplied into bigger messes, faster. A harness doesn't guarantee a positive outcome — it increases the _probability_ the multiplier lands positive. It does this two ways: **guardrails** (the governance skills' AUDIT/CONFORM gates, the checker contract, adversarial safety-review for auto-executing surfaces) constrain the downside tail; **tools** (skills, agents, MCP servers) are targeted upgrades — each governed surface acts like a named enchanted item with a specific bonus against a specific failure mode (e.g. `ki-decision-records`' audit as a "+2 sword of justice" against untraceable decisions), not a generic buff. Worth stating explicitly alongside the vendoring-framing item above, as the _why_ to that item's _how_.

### Vendor skills into `.ki-meta/skills`, then symlink `.claude/skills` from there — no symlink target outside the repo _(candidate)_

Today `ki:skills:link:project` (`skills/keystone/ki-bootstrap/scripts/link-skills.ts`) symlinks a consumer repo's `.claude/skills/<skill>` straight at this harness's own `skills/<cluster>/<skill>` — a relative symlink that resolves **outside the consuming repo entirely**, onto wherever this harness happens to be checked out locally. That is a different pattern from how checkers are already handled: `bootstrap.ts` vendors (copies) each declared skill's `audit.ts`/`conform.ts` into the repo's own `.ki-meta/skills/<skill>/`, precisely so the repo is self-governing without depending on an external checkout path — the same reason installed scripts must be copies, not symlinks (rubric `SCRIPT-7`). Apply the same discipline one level up: vendor the **whole skill directory** into `.ki-meta/skills/<skill>/`, then have `.claude/skills/<skill>` symlink to that in-repo copy instead of reaching across the filesystem to this harness's checkout. Generalises to anything else vendored (agents, hooks): the in-repo `.ki-meta/` copy is the source of truth a symlink may point to; a symlink should never leave the repo boundary. Needs design: how whole-skill content and the existing checker files coexist in the same `.ki-meta/skills/<skill>/` namespace, how BOOT-11 expands beyond direct audit/conform units, how `link-skills.ts`'s `--check`/orphan/broken-link detection changes when the symlink target is local, and whether this changes the re-vendor workflow described in this repo's own `CLAUDE.md`.

### A thin `ki` command-line tool wrapping the vendored scripts _(candidate)_

Requested 2026-07-15: give the harness a globally-installed command-line tool the way [Headroom](https://extraheadroom.com) ships one — a short `ki` command on `PATH` — rather than requiring per-repo invocation. Today a governed repo self-governs two ways, both awkward for a user at a shell: `bun run ki:audit`/`ki:conform` (only where the repo carries the `ki-engineering` toolchain and a `package.json` with those keys) or `./.ki-meta/bin/ki-audit` (works package.json-free, but the user must be at the repo root and know the vendored path). A `ki audit` / `ki conform` on `PATH` would walk up from the current directory to the nearest `.ki-meta/bin/` and `exec` its wrapper — so any bootstrapped repo audits or conforms with one short command from anywhere in its tree, regardless of whether it has a `package.json`. Keep it **super-thin**: no logic of its own, purely discovery-plus-delegation to the vendored `aggregate.ts`, exactly mirroring how `.ki-meta/bin/ki-audit` already `exec bun`s it — the tool must never reimplement or drift from the vendored checkers, only locate and hand off to them. Natural fit for the `ki-tools` standard (one standalone CLI per repo, `curl | bash` installer + Homebrew tap formula) as a new `tools-ki` repo, distributed alongside `tools-mgit`. Needs design: how it reconciles with the two existing invocation paths (pairs with the "Explicitly test the `.ki-meta/bin/ki-audit`/`ki-conform` shell wrapper against the `bun run ki:*` path" item under **Soon** — the tool becomes a third form that must produce identical exit codes and findings); whether it lives as a `tools-*` repo or ships from the harness itself; and how upward `.ki-meta/bin/` discovery behaves across nested or sibling governed repos.

### Port the KI MCP servers into Cowork's sandbox

The `ki-binding` cross-surface skill is built, and the Cowork leg ships the `knowledgeislands/ki-plugins` marketplace plugin — but **skills + agents only**. The KI MCP servers are host-local (they read host filesystem paths and resolve secrets via 1Password) and Cowork runs plugins in a gVisor sandbox that cannot reach the host, so the servers do not port as-is (verified 2026-07-06, [cross-surface-enablement.md](../../../skills/repo-structure/ki-mcp/references/cross-surface-enablement.md) Verification log). Making them reachable needs one of: a self-contained server bundled via `${CLAUDE_PLUGIN_ROOT}`, mounting the KB into the sandbox, or authenticated remote (`http`/`sse`) endpoints. Then `build-plugin.ts` emits a `.mcp.json` and the `cowork` `clients` token becomes live. Related open question: `ki-plugins` keeps the harness's proprietary LICENSE while public, which conflicts with `ki-repo`'s public-repo-MIT check — resolve the licensing/visibility stance. Separately, Cowork supports a browser-control plugin capability that Claude Code's harness currently has no equivalent for (unverified against current docs as of 2026-07-14) — worth confirming what that capability actually offers before deciding whether it changes the shape of this port.

### Run claude.ai connectors' equivalents behind mcporter _(candidate)_

Several claude.ai managed connectors have a KI-owned local equivalent already in the estate — Google Calendar/Drive/Gmail ↔ `mcp-gsuite`, Microsoft 365 ↔ `mcp-m365` — so the same capability could be served from the local server set behind mcporter rather than each surface authorising a separate hosted connector. This lands cleanly on the **local** surfaces (Claude Code, Desktop): the server sits behind the single `ki-mcporter` slot ([ADR-KI-HARNESS-TOOLCHAIN-003](../../../docs/decisions/ADR-KI-HARNESS-TOOLCHAIN-003-mcporter-mcp-proxy.md)) and Tool Search keeps its schemas off the standing surface until used. The blocker is the same host-local reachability wall as the Cowork item above: **claude.ai (web) cannot reach a localhost daemon**, so parity there needs an authenticated public endpoint (a tunnel + auth) registered as a custom connector — a deliberate security decision — or accepting that web keeps the hosted connector while local surfaces use the mcporter-fronted server. Investigate the mapping (which connectors have a local equivalent worth running, which are better left hosted — Slack/Linear/Attio/Xero/Granola have no KI server), the tunnel/auth posture, and how `ki-binding`'s `clients:` targeting expresses "local surfaces use the server, web uses the connector" without the surfaces silently diverging.

### Run Hermes remotely for background asynchronous work _(candidate)_

[Hermes](https://github.com/nousresearch/hermes-agent) is a candidate host for offloading long-running or overnight agent work — the goal is background async execution (e.g. while asleep or away from the laptop), not a replacement for the interactive harness workflow; keep Hermes a tool invoked for a specific class of work, not the main focus of daily use. Two candidate homes: the Mac Studio (always-on, local network, no extra cost) versus an AWS account (reachable from anywhere, but adds cloud spend and a security/access-control surface to manage). Evaluate: which harness surfaces (skills/agents/MCP) Hermes needs to reach and whether that repeats the host-local reachability wall already tracked above (Cowork sandbox, mcporter web-parity items) — a remote Hermes instance likely needs the same authenticated-endpoint treatment as those; how a job gets handed to Hermes and how its results get back into the harness/KB; and what "done, needs review" surfacing looks like so background runs don't silently pile up unreviewed.

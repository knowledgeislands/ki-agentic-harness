# ki-agentic-harness roadmap

Where this agentic harness is going. The [README](README.md) and the [docs/](docs) it indexes cover what exists today and how to install it; this file is the forward view. Open work is grouped by horizon ā€” **Next**, **Soon**, **Future** ā€” the house phasing owned by `ki-harness` and referenced by `ki-plans`; speculative items are marked _(candidate)_ until committed.

This roadmap is itself subject to the house discipline it describes: when a skill's REFRESH run or an audit surfaces a structural gap, it lands here before it's built ā€” and an item is **removed once done**, not ticked off, so the file always shows only open work.

**Continuous practices are not roadmap items.** Keeping the skills audited (`ki:skills:lint`, `ki:repo:audit`, `ki:kb:audit`, the `ki-mcp` audit over the `mcp-*` repos), re-running the advisory [eval suite](evals/README.md) as skills change, and the scheduled `ki-skills-refresh` sweep (which honours each skill's declared `**Refresh:**` cadence) are ongoing disciplines tied to the invariants in [docs/skill-design.md](docs/guides/user-guide/skill-design.md) (_Principles across the set_) ā€” they run continuously, so they live there, not here.

## Soon

### Codify a Conventional Commits git standard across the skills

The repos already commit in [Conventional Commits](https://www.conventionalcommits.org/) style (`type(scope): subject`, seen throughout the history), but nothing in the skill set **governs** it ā€” there is no house standard a commit is audited or conformed against. Land one: decide which skill owns git conventions (candidate: `ki-engineering`, which already owns the CI-workflow shape and the toolchain, or a dedicated `ki-git` if the surface is broad enough ā€” commit format, branch naming, the solo-repo-to-`main` policy), specify the allowed `type` set and scope conventions, and add a mechanical check (e.g. a commit-message linter wired into the husky hook chain) plus the AUDIT/CONFORM prose. Cross-reference the per-repo commit guidance already living in `CLAUDE.md` files so the standard and the local instructions stay consistent.

### Normalise mode vocabulary and mode-heading structure across the skills

Building the HELP mode ([ADR-KI-HARNESS-SKILLS-001](docs/decisions/ADR-KI-HARNESS-SKILLS-001-canonical-modes.md)) surfaced that the skills disagree on two things the `skill-help.ts` renderer had to be made tolerant of, and which the `ki-skills` rubric does not yet enforce. **Mode verbs drift:** the canonical scaffold mode is `INIT`, but `ki-decision-records`, `ki-kb-activities`, and `ki-kb-live-artifacts` use `NEW`, and `ki-feature-definitions` carries _both_ ā€” decide whether `NEW` is an alias to retire in favour of `INIT` or a distinct mode worth defining, and settle the accepted skill-specific extensions (`OPTIMISE`, the `ki-kb-streams` lifecycle verbs) against the universal set. **Mode-heading structure drifts:** most skills use `## Mode X ā€” gloss` (H2), `ki-authoring` uses `### Mode X` (H3), and `ki-kb` / `ki-housekeeping` declare their modes in prose with no per-mode heading at all ā€” pick one convention so a mode list is mechanically extractable the same way everywhere (rather than relying on the renderer's argument-hint fallback). Land both as `ki-skills` rubric criteria with a mechanical check in `lint-skills.ts`, then conform the set.

### Cover the CLAUDE.md ā†’ skill move as a tokenomics lever

The [tuning guide](docs/guides/user-guide/tuning.md) documents three ways to lighten the prose layer ā€” trim `CLAUDE.md`, apply the layer scope rule (repo-specific ā†’ repo `CLAUDE.md`, cross-project ā†’ `~/.claude/*.md`, facts ā†’ memory), and reconcile a promoted memory by deleting it ā€” but not the `CLAUDE.md` ā†’ **skill** move. `CLAUDE.md` is always-on standing surface; a skill's _body_ loads **on demand**, so only its ~50-token `description` stays in the prefix. A chunk of `CLAUDE.md` that is only _situationally_ relevant (an occasional procedure, a domain playbook) is therefore cheaper as an on-demand skill: pay the description every turn, defer the body until a task triggers it. Add a short paragraph to the guide's trim section drawing this always-on-vs-on-demand trade-off, with the caveat that only genuinely situational, self-contained content should move (lifting a one-liner costs more in standing description than it saves), and route the "is this skill-worthy" call to `/ki-skills` judgment rather than prescribing it. Consider whether it also earns a `ki-tokenomics` rubric point (progressive disclosure of the prose layer) alongside the guide prose.

### Check naming consistency across the `.ki-meta/` migration

The self-sufficiency relocation moved the vendored checkers from `scripts/ki/` to `.ki-meta/` with a `.ki-meta/bin/ki-audit` wrapper (no `package.json` required). That reopens the naming-consistency question that started as "should `ki:audit`/`ki:conform` carry a distinguishing marker": now the names span **three** surfaces that must agree ā€” the `.ki-meta/` script paths and directory layout, the `bin/` wrapper name(s), and the `package.json` `ki:*` script targets that still front them where a repo has a `package.json`. Audit that the aggregate-vs-scoped distinction (2-segment `ki:<verb>` = repo-wide entrypoint, 3-segment `ki:<slug>:<verb>` = skill-scoped ā€” the decision taken this session was to keep it) is expressed **consistently** across all three: the `.ki-meta/bin/ki-audit` wrapper name, the `.ki-meta/bin/aggregate.ts` verb dispatch, and the `package.json` keys don't drift or collide. Decide whether the convention earns an explicit stated rule in the `ki-engineering` standard (currently prose + segment count only, not mechanically enforced) and whether the `bin/` wrapper naming needs its own line. Sweep the 6 MCP repos too ā€” they are committed on the **old** `scripts/ki/` layout and will need re-bootstrapping onto `.ki-meta/`, at which point their `package.json` `ki:*` targets must be re-checked against the new wrapper.

### Wire the `ki:*` package.json convenience keys from `ki-engineering`

Bootstrap builds `.ki-meta/` and writes the four `./.ki-meta/bin/{ki-audit,ki-conform,ki-init,ki-help}` wrappers, but deliberately touches **no `package.json`** ā€” the split locked this session (ADR-KI-HARNESS-006): those bins are the canonical interface, and the `ki:*` keys a code repo may want are additive sugar over them, owned by `ki-engineering`. Land that in `ki-engineering`: its INIT scaffolds the repo-wide aggregate keys (each simply `bun ./.ki-meta/bin/ki-<verb>`) and its AUDIT checks they front the vendored bins without drifting. This also settles the `ki:conform` name overlap ā€” `ki-engineering`'s existing `ki:conform` is its lint write-pass, so the bootstrap aggregate stays the bin, and any repo-wide key ki-engineering adds must not collide (a distinguishing segment, or leaving self-governance to the bins). Reconcile with the naming-consistency sweep above.

### Author the missing `conform-*.ts` scripts across the fleet

The `.ki-meta` vendoring mechanism carries any per-skill mechanical script ā€” CONFORM vendors wherever a skill ships a `conform-*.ts`, and `./.ki-meta/bin/ki-conform` fans out over whatever is present ā€” but today only `ki-repo`, `ki-binding`, `ki-decision-records`, and `ki-bootstrap` ship one, so conform is thin fleet-wide. Work skill-by-skill deciding what each skill's CONFORM can fix _deterministically_ (the judgment is per-skill: some conform steps are inherently LLM/judgment and stay out of the vendored script), authoring a `conform-*.ts` for each that has a mechanical floor, with 100% test coverage per `ki-engineering`. The INIT and HELP threads of this work are now shipped ā€” the aggregate `init` verb re-syncs via the manifest drift contract, and HELP vendors a rendered per-skill snapshot that `ki-help` serves as pure bash ā€” so CONFORM authoring is the remaining body-of-work.

### Reconcile the harness's own `.ki-meta/` with the documented bin surface

`onboarding.md` and ADR-KI-HARNESS-006 describe bootstrap building four `./.ki-meta/bin/{ki-audit,ki-conform,ki-init,ki-help}` wrappers plus a `.ki-meta/manifest.json`, but the harness's own `.ki-meta/bin/` currently holds only `aggregate.ts` and `ki-audit`, with no manifest. Either the harness has not been re-bootstrapped onto the current chain, or the surface narrowed. Re-run bootstrap (or correct the guide) so the harness's self-hosted `.ki-meta/` matches what its docs promise ā€” the onboarding guide claims these blocks are test-exercised and cannot drift, so the gap is notable. A stale-**content** dimension now compounds the stale-**surface** one: the four checkers were conformed to the pinned `--json` shape in `skills/` (the `ki-housekeeping`/`ki-binding`/`ki-decision-records`/`ki-feature-definitions` audits), but their vendored `.ki-meta/skills/*/audit.ts` copies still emit the old shapes, and BOOT-9's set-based check does not flag content drift ā€” so a faithful resync is the deliberate re-bootstrap, not an incidental edit (see the `.ki-meta` re-vendor caution in `CLAUDE.md`). **Unblocks:** once the vendored checkers emit the pinned wrapper, `aggregate.ts`'s shape-detection fallback (grown to survive `ki-housekeeping`'s bare-array output) becomes dead weight and can be simplified back to a single wrapper-object read path.

### Codify the generated-code lint/knip exclusion in `ki-engineering`

ADR-KI-HARNESS-TOOLCHAIN-005 records that generated and vendored code (`src/generated/`, the vendored-checker tree) is excluded from Biome and knip ā€” applied per-repo in config today, not stated in the standard. Lift it into `ki-engineering` so its AUDIT expects the exclusions (a generated tree present but not excluded is a finding) and its exemplars show the `!`-negated `files.includes` and the `knip.ignore` entries.

## Future

### Reclassify the remaining borderline DRs _(candidate)_

The main reclassification landed: [GDR-KI-HARNESS-001](docs/decisions/GDR-KI-HARNESS-001-adopting-decision-records.md) (adopting DRs), the [SDR-KI-HARNESS-001](docs/decisions/SDR-KI-HARNESS-001-purpose-and-scope.md) / [ADR-KI-HARNESS-001](docs/decisions/ADR-KI-HARNESS-001-repository-structure.md) split (purpose/scope vs structure), and [GDR-KI-HARNESS-002](docs/decisions/GDR-KI-HARNESS-002-public-and-declared-license.md) (license) now carry their true prefixes. What remains is a small set of genuinely borderline records the sweep deliberately left as `ADR`: `ADR-KI-HARNESS-003` (mechanical-first reads governance-flavoured), `ADR-KI-HARNESS-TOOLCHAIN-002` (an adopt/decline tooling survey, closer to strategy), and `ADR-KI-HARNESS-SKILLS-007` (ki-housekeeping, closer to operations). Each is a judgment call with a weak case for retyping; revisit only if the ambiguity starts to mislead.

### Sweep decisions, feature definitions, and guides for drift and stale citations _(candidate)_

Each governance skill's own REFRESH mode checks its **standard** against its sources, and the scheduled `ki-skills-refresh` sweep honours that cadence ā€” but nothing checks the **content written under those standards** for staleness or cross-artifact drift: whether a DR's `## References` still point at sibling records that exist and haven't been retitled, whether a feature definition's cited decisions still hold, whether a guide in `docs/guides/` still describes what the skills actually do today. `ki-decision-records`'s own mechanical checker (`audit-drs.ts`) already validates DR structure and index completeness within `docs/decisions/`, but stops at that directory's boundary ā€” it doesn't reach into `docs/features/` or `docs/guides/` to check the links and claims that cross between them. Investigate a periodic sweep ā€” mechanical where citation existence can be checked by a script (dead internal links, missing index entries), judgment-based where content currency is the question (does this guide's description still match the skill it documents) ā€” scoped to run less often than per-commit, more like the `ki-skills-refresh` cadence. Decide ownership: a natural extension of `ki-decision-records`' own REFRESH, a `ki-feature-definitions` counterpart, or a cross-cutting sweep skill that walks all three surfaces in one pass.

### Roll out the expanded Feature Definitions pattern to the fleet _(candidate)_

The harness's `docs/features/` now covers the five-part bundle, the universal modes, and the checker contract as a checkable RFC-2119 inventory ā€” a worked exemplar of `ki-feature-definitions`. Roll the same pattern out to the sibling repos (the `mcp-*` servers, `ki-website`, the KB), each getting a small `docs/features/` describing what that repo does and checkable against its own behaviour, using the harness set as the template.

### Hooks as a governed harness surface ā€” starting with the plan-file lifecycle _(candidate)_

The harness governs skills, agents, MCP servers, and evals ā€” its four-part layout owned by `ki-harness` ā€” but **not hooks**, its own deterministic automation surface (the `settings.json` `PreToolUse` / `PostToolUse` / `Stop` / `SessionStart` command hooks the runtime fires). Today those are unmanaged: the working hooks live in personal chezmoi dotfiles (`chezmoi-drift.sh`, `git-lock-check.sh`), invisible to the skill set and un-audited. The motivating first use is a **plan-file lifecycle pair**: a `PostToolUse(ExitPlanMode)` hook that stamps `status` / `created` / `cwd` frontmatter onto every new plan ā€” so plan cleanup becomes greppable rather than inferential, the exact gap that prompted this item ā€” and a `PostToolUse(TodoWrite)` hook that mirrors live todo state into the plan and flips `status: open ā†’ done` when all todos complete, riding on the todos the model already maintains. Both are specced and ready (BSD-`sed`-safe bash via awk + atomic writes, `jq`-parsed and injection-safe, exit-0-always; the `ExitPlanMode` payload was found to carry `.tool_input.planFilePath`, so the target file is unambiguous). They ship first into chezmoi because hook _activation_ is a single monolithic `settings.json` block that cannot be symlinked the way `~/.claude/skills|agents/` are directory-discovered ā€” so chezmoi owns registration no matter where the scripts live. The roadmap question is whether hooks should also become a **harness-owned surface**: a top-level `hooks/` part-dir (parallel to `agents/`) plus a `ki:hooks:link` linker cloned from `scripts/sync-skills.ts`, and possibly a `ki-hooks` skill (or a `ki-harness` extension) that AUDITs/CONFORMs registered hooks the way the other surfaces are governed. **Blocked** on the in-flight linker / `package.json`-key rewrites (`link-skills.ts` `cmdLink`/`cmdCheck`), with which a `ki:hooks:link` would directly conflict ā€” sequence after they land; the swap is non-breaking because the registration path (`$HOME/.claude/hooks/*.sh`) is stable whether the file is a chezmoi copy or a harness symlink. Cross-cuts the `ki-recap` process skill, which already names "a hook" as a destination for harvested learnings ā€” a governed hooks surface is what makes that routing real. Scope boundary to hold: personal-global workflow hooks (plan hygiene) stay in chezmoi; only hooks that should travel with a repo to any contributor are candidates for harness ownership.

### Mine historical sessions for recurring context bloat _(candidate)_

Common issues tend to cost the same context over and over ā€” an audit script that scans all repo Markdown and returns spurious FAILs, a Read tool that fails silently on space-containing paths, a checker invoked with the wrong default argument ā€” each rediscovered from scratch across sessions before the fix is known. The `headroom learn` sweep already captures some of these as _Learned Patterns_ (see this repo's `CLAUDE.md`), but that is a manual, per-session distillation. Investigate whether a **mechanical tool** could analyse the corpus of historical session transcripts (the stored VSCode/Claude Code sessions the `ki-housekeeping` standard already governs) to detect repeated, high-cost context patterns ā€” the same dead-end tool call, the same re-read of a large file, the same clarification round-trip ā€” and surface them as candidates for a `CLAUDE.md` learned-pattern entry, a skill fix, or a hook. This is the **offline, cold** sibling of the `ki-recap` process skill (which already ships the **warm, in-session** leg): the two share the transcript-reading substrate and the routing table, so the open design question is not "one tool or two" (answered: two) but how much of this one can reuse `ki-recap`'s `scripts/recap-grounding.ts` versus needing its own mechanical corpus-scan. Assess overlap with `headroom learn` and `ki-tokenomics` (which owns the standing context surface but not this retrospective, per-issue analysis), whether the signal is extractable without an LLM pass, and where such a tool would live (candidate: a `ki-housekeeping` companion, since it already has session-transcript access).

### Evaluate memtrace as structural-memory layer for the harness _(candidate)_

[syncable-dev/memtrace-public](https://github.com/syncable-dev/memtrace-public) is a code-intelligence tool that parses a codebase into a live knowledge graph ā€” symbols as nodes, relationships (`CALLS`, `IMPLEMENTS`, `IMPORTS`) as edges, built with Rust + Tree-sitter ā€” that agents query in milliseconds. It claims deterministic indexing (no LLM calls, no API cost), bi-temporal version history with scoring modes for "what changed / what might break" queries, cross-service HTTP call-graph topology, and ships 25+ MCP tools and 17 agent skills across 20+ languages. Currently private beta under a proprietary EULA (benchmark suite MIT). Assess whether it complements the harness ā€” as a structural-memory MCP server the KI skills could lean on, or as prior art for the skills/agents/MCP bundle shape ā€” and whether its licensing and host-local posture fit our surfaces.

### Port the KI MCP servers into Cowork's sandbox

The `ki-binding` cross-surface skill is built, and the Cowork leg ships the `knowledgeislands/ki-plugins` marketplace plugin ā€” but **skills + agents only**. The KI MCP servers are host-local (they read host filesystem paths and resolve secrets via 1Password) and Cowork runs plugins in a gVisor sandbox that cannot reach the host, so the servers do not port as-is (verified 2026-07-06, [cross-surface-enablement.md](skills/ki-mcp/references/cross-surface-enablement.md) Verification log). Making them reachable needs one of: a self-contained server bundled via `${CLAUDE_PLUGIN_ROOT}`, mounting the KB into the sandbox, or authenticated remote (`http`/`sse`) endpoints. Then `build-plugin.ts` emits a `.mcp.json` and the `cowork` `clients` token becomes live. Related open question: `ki-plugins` keeps the harness's proprietary LICENSE while public, which conflicts with `ki-repo`'s public-repo-MIT check ā€” resolve the licensing/visibility stance.

### Run claude.ai connectors' equivalents behind mcporter _(candidate)_

Several claude.ai managed connectors have a KI-owned local equivalent already in the estate ā€” Google Calendar/Drive/Gmail ā†” `mcp-gsuite`, Microsoft 365 ā†” `mcp-m365` ā€” so the same capability could be served from the local server set behind mcporter rather than each surface authorising a separate hosted connector. This lands cleanly on the **local** surfaces (Claude Code, Desktop): the server sits behind the single `ki-mcporter` slot ([ADR-KI-HARNESS-TOOLCHAIN-003](docs/decisions/ADR-KI-HARNESS-TOOLCHAIN-003-mcporter-mcp-proxy.md)) and Tool Search keeps its schemas off the standing surface until used. The blocker is the same host-local reachability wall as the Cowork item above: **claude.ai (web) cannot reach a localhost daemon**, so parity there needs an authenticated public endpoint (a tunnel + auth) registered as a custom connector ā€” a deliberate security decision ā€” or accepting that web keeps the hosted connector while local surfaces use the mcporter-fronted server. Investigate the mapping (which connectors have a local equivalent worth running, which are better left hosted ā€” Slack/Linear/Attio/Xero/Granola have no KI server), the tunnel/auth posture, and how `ki-binding`'s `clients:` targeting expresses "local surfaces use the server, web uses the connector" without the surfaces silently diverging.

Two leads to check when this is picked up: mcporter itself already covers some of the no-KI-server set ā€” [openclaw/mcporter#67](https://github.com/openclaw/mcporter/issues/67) discusses Granola handling, and Linear appears in mcporter's example docs ā€” so those may need a mcporter config, not a new KI server. Separately, revisit how `mcp-gsuite` and `mcp-m365` do OAuth: if mcporter (or the wider MCP ecosystem) has since landed a cleaner pattern for OAuth-backed MCP servers than the current host-local 1Password-resolved flow, prefer it ā€” that could also lower the reachability/auth bar for the web-parity question above.

### Run Hermes remotely for background asynchronous work _(candidate)_

[Hermes](https://github.com/nousresearch/hermes-agent) is a candidate host for offloading long-running or overnight agent work ā€” the goal is background async execution (e.g. while asleep or away from the laptop), not a replacement for the interactive harness workflow; keep Hermes a tool invoked for a specific class of work, not the main focus of daily use. Two candidate homes: the Mac Studio (always-on, local network, no extra cost) versus an AWS account (reachable from anywhere, but adds cloud spend and a security/access-control surface to manage). Evaluate: which harness surfaces (skills/agents/MCP) Hermes needs to reach and whether that repeats the host-local reachability wall already tracked above (Cowork sandbox, mcporter web-parity items) ā€” a remote Hermes instance likely needs the same authenticated-endpoint treatment as those; how a job gets handed to Hermes and how its results get back into the harness/KB; and what "done, needs review" surfacing looks like so background runs don't silently pile up unreviewed.

Bring the repo's directory/package name in line with the `ki-` prefix convention. Mechanical rename across the repo (package.json `name`, README, CI badges, any cross-repo references such as `ki-agentic-harness/scripts/generate-clients.ts` and the `ki-mcp` standard doc's repo list) plus the GitHub repo rename itself.

### The harness `docs/` is canonical authoring; the website vendors a copy

Skill and design documentation is authored here and stays canonical ā€” it is an **input** to two downstream consumers: ki-arcadia-principal references it (via the `Agentic Tool Documentation` stream), and ki-website needs it for its pages. The harness stays **canonical for its own story** ā€” the human-first overview ([docs/overview.md](docs/guides/user-guide/overview.md)) and the skill/design docs live here; the downstream repos reference rather than own them.

Because the website build must be **autonomous** ā€” no reach into a sibling repo at build time ā€” it cannot source the harness `docs/` live; it vendors a self-contained copy. Automate that as a **one-directional committed sync**: a script or CI step in ki-website copies the website-relevant subset of the harness `docs/` into its content tree and commits it, so the build is fully self-contained, the harness stays the single source, and every update lands as a reviewable diff in the website's history ā€” no manual duplication. (A git submodule pinned to a harness ref is the alternative, if strict version-pinning is preferred over copy-commits.)

**Gate:** this pipeline is ki-website's roadmap work; the harness's only responsibility is to remain the canonical author. Nothing here for the harness to retire.
